# SiligenSuite 阶段 0 基线盘点

## 1. 目的

本文件用于在正式重构前，先把旧仓 `D:\Projects\Backend_CPP` 的真实系统边界盘清楚。

阶段 0 不做大规模目录移动，只做三件事：

1. 盘点运行入口
2. 盘点契约与协议边界
3. 盘点领域能力

这三张表将作为后续迁移顺序、拆包边界和测试策略的基线。

---

## 2. 状态标记说明

- `已独立迁出`：已经从旧仓拆成独立子项目
- `迁移中`：已有新位置，但 legacy 路径或兼容壳仍然存在
- `待补位`：当前工作区里还没有形成清晰的一等位置
- `新增`：旧仓没有独立存在，但当前工作区新增了此能力

### 4.1 当前最需要先冻结的四条契约

后续重构开始前，最应该先冻结版本和兼容策略的是：

1. HMI <-> Core 应用协议
2. DXF `.pb` / JSON 输出协议
3. HMI <-> DXF Editor 的 CLI + notify 协议
4. Core <-> Device 的设备能力契约

如果这四条边界不先明确，后面拆包会不断回滚。

## 6. 当前阶段的直接结论

### 6.1 当前工作区已经拆出的能力

已经较明确拆出的有：

- HMI
- DXF 编辑器
- DXF 预处理管线
- 仿真引擎
- 控制核心中的 TCP server 与 runtime 入口

### 6.2 当前仍未形成清晰一等位置的能力

以下能力虽然存在，但还没有完全形成清晰的一等工程边界：

- CLI
- 设备能力契约
- 配方数据与 schema 的正式归属
- 报警/状态/工艺结果查询契约
- 配置契约的统一治理

### 6.3 后续迁移优先级建议

建议优先顺序：

1. 先冻结契约
2. 再明确应用入口
3. 再收拢核心业务内核
4. 最后剥离宿主、传输和设备实现

这个顺序比直接移动目录更稳。

## 7. 阶段 0 完成标准

满足以下条件，才算阶段 0 完成：

1. 所有一等运行入口都有明确归属
2. 所有跨边界协议都有 owner、版本策略和兼容性测试位置
3. 所有领域能力都能映射到目标工作区中的明确归宿
4. 迁移中能力和待补位能力被显式标记，而不是靠口头记忆
