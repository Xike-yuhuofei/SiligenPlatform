# SiligenSuite 整体重构方案（修订版）

## 1. 方案定位

本方案不是仅基于当前 `SiligenSuite` 已拆出的几个子项目整理目录，
而是以 `D:\Projects\Backend_CPP` 的真实系统全貌为基线，
重新给出一版适合长期演进的整体重构方案。

本方案要解决的问题不是“如何把目录摆整齐”，而是：

1. 如何把旧仓中混合在一起的运行入口、领域逻辑、硬件适配、工具链和数据契约重新分层
2. 如何在不牺牲核心链路协同效率的前提下，建立清晰、可执行、可测试的边界
3. 如何让当前迁移中的多个子项目，最终收敛为一个统一治理的工程，而不是继续并排演化

---

## 2. 旧仓基线：Backend_CPP 的真实能力版图

在 `Backend_CPP` 中，当前系统并不是单一“控制核心”，而是一个历史上逐步生长起来的综合仓。

### 2.1 运行入口

- CLI：`siligen_cli.exe`
- TCP 服务：`siligen_tcp_server.exe`
- HMI：`src/adapters/hmi/main.py`
- DXF 工具链：仓内 `tools/dxf_pipeline`，并支持外部 `D:\Projects\DXF`

### 2.2 领域层能力

旧仓领域层包含以下长期存在的核心能力：

- `motion`
- `dispensing`
- `recipes`
- `machine`
- `safety`
- `diagnostics`
- `trajectory`
- `planning`
- `geometry`
- `configuration`
- `system`

这些能力并不是临时实现细节，而是后续重构必须显式建模的长期能力。

### 2.3 基础设施与宿主能力

旧仓还包含以下系统级能力：

- 硬件驱动与控制卡适配
- 配置加载与资源文件
- 启动装配与依赖注入容器
- 事件/日志/记录持久化
- 安全相关支撑
- Python HMI 与 C++ 运行时之间的 TCP 协议

### 2.4 工具与外围应用

- DXF 编辑器
- DXF 预处理管线
- 轨迹比较/分析工具
- 配方数据目录与结构化数据资产

结论：
`Backend_CPP` 的真实系统边界比当前 [重构.md](D:/Projects/SiligenSuite/重构.md) 里描述的范围更大。
因此，重构方案必须覆盖“应用入口 + 核心业务 + 系统宿主 + 契约 + 工具链”五个层面。

## 3. 修订后的总判断

最终形态仍然建议采用：

**单仓多库（Monorepo + Multi-Package + Multi-App）**

但需要在原方案上做四个关键修订：

1. 不能只保留 `hmi-app` 一个应用入口，必须把 `control-runtime`、`control-tcp-server`、`control-cli`、`dxf-editor-app` 一并纳入整体结构
2. 不能只建“业务包”和“设备包”，必须显式建模“运行时宿主/装配层”
3. 不能把跨语言协作简单理解为共享代码，必须把契约沉淀为 schema/codegen/doc，而不是某一端内部类
4. 不能只按当前拆出的项目命名包，必须覆盖旧仓长期存在的 `recipes / machine / safety / diagnostics / configuration` 等领域能力

因此，推荐的最终方向不是“原 `重构.md` 原样执行”，而是：

**单仓多库 + 多应用入口 + 核心链路收拢 + 系统宿主独立 + 契约显式化 + 跨语言契约治理 + 集成验证一级化**

## 4. 重构原则

### 4.1 一级领域不等于一级包

认知上可以有：

- 工程数据
- 工艺与执行
- 机器状态与安全
- 配方
- 设备能力
- 追溯与诊断
- HMI
- DXF 编辑
- 仿真
- 运行时宿主

但工程落地时，不必把每个领域都物理拆成独立包。
要根据协同强度、接口成熟度和测试方式决定包边界。

### 4.2 核心链路优先收拢

当前最不适合过早拆散的仍然是：

```text
工程对象/工艺意图 -> 规划 -> 执行 -> 机器状态 -> 异常恢复
```

因此 `process-runtime-core` 仍应作为强收拢内核存在，
但必须在包内显式区分子域，而不是继续演化成新的巨石。

### 4.3 运行时宿主要单独建模

旧仓中的容器装配、配置加载、资源路径、安全初始化、后台监控等能力，
既不是纯业务，也不是纯设备驱动。
它们应从业务核心中分离出来，形成独立的 `runtime-host`。

### 4.4 跨语言边界必须契约先行

HMI 是 Python，主控核心和仿真大部分是 C++。
因此：

- `application-contracts` 不能只是一套 C++ DTO
- `engineering-contracts` 不能只是一套 Python 数据类

契约包应以 schema、IDL、proto、JSON schema、codegen、版本策略和兼容性测试为主。

### 4.5 测试分三层治理

必须区分：

1. 包内单元测试
2. 跨包集成测试
3. 仿真/硬件联调/场景回归

工业控制软件不能只靠单元测试判断质量。

## 6. 顶层结构说明

### 6.1 `apps/`

放最终可运行入口，不放可复用能力。

应至少包含：

- `control-runtime`
- `control-tcp-server`
- `control-cli`
- `hmi-app`
- `dxf-editor-app`

这五个入口都在旧仓中真实存在或已有长期使用语义，
不应只把 HMI 视为“应用”，而把其余入口隐藏在核心包内部。

### 6.2 `packages/`

放核心复用能力、契约、宿主支撑和公共基础模块。
这是 monorepo 的真正核心区域。

### 6.3 `integration/`

放跨包协同验证。
需要显式覆盖：

- 工程数据到规划
- 规划到执行
- HMI 到 TCP
- 契约兼容性
- 仿真回归
- 硬件冒烟

### 6.4 `data/`

旧仓中的配方数据、schema、审计和回归样本是系统资产的一部分，
不应被散落在各个应用或包中。

## 7. 包级方案

## 7.1 `packages/engineering-data`

### 定位

承接旧仓中的 `geometry / trajectory / planning` 里偏工程对象与几何预处理的一部分，
以及当前 `dxf-pipeline` 中稳定可复用的工程数据能力。

### 负责内容

- DXF 导入、解析与标准化
- 几何模型与拓扑关系
- 坐标、方向、单位标准化
- 工程对象合法性校验
- 面向核心链路输出稳定工程对象模型

### 不负责内容

- 任务执行
- 设备控制
- UI 编辑交互
- 真实运行时状态机

### 说明

`engineering-data` 不等于整个 `dxf-pipeline`。
`dxf-pipeline` 中与版本化输出格式相关的部分，应与 `engineering-contracts` 协作治理。

---

## 7.2 `packages/process-runtime-core`

### 定位

系统业务内核，收拢旧仓中最强协同演化的领域能力。

### 必须覆盖的内部子域

- `process`
- `planning`
- `execution`
- `machine`
- `safety`
- `recipes`
- `dispensing`
- `diagnostics-domain`
- `application`

### 负责内容

- 工艺规则与制造语义
- 规划与执行编排
- 任务系统、状态机、暂停恢复
- 机器运行模式
- 安全互锁规则
- 配方生效规则
- 点胶执行规则
- 工艺结果的领域聚合与校验
- 面向外部提供统一应用服务

### 不负责内容

- 控制卡驱动实现
- TCP 会话与网络协议
- HMI 页面与 UI 状态
- 宿主初始化与资源装配

### 关键说明

这里的 `diagnostics-domain` 是领域规则，不是日志平台。
如果不把这层规则单独说明，后续很容易把旧仓里的诊断结果模型错误地塞进“纯日志模块”。

---

## 7.3 `packages/runtime-host`

### 定位

承接旧仓中的容器装配、配置加载、资源初始化、运行时宿主、后台任务和系统级启动逻辑。

### 负责内容

- 依赖装配与容器配置
- 配置加载与环境切换
- 资源路径管理
- Runtime 宿主生命周期
- 后台监控启动
- 安全与系统级初始化

### 不负责内容

- 业务规则
- UI
- 控制卡具体驱动
- TCP 协议细节

### 为什么必须独立

如果不单列 `runtime-host`，
这些能力最终会重新回流进 `process-runtime-core` 或入口应用，
导致边界再次混乱。

---

## 7.4 `packages/device-adapters`

### 定位

旧仓硬件适配层与当前 `device-hal` 的演化归宿。

### 负责内容

- 控制卡适配
- 运动控制能力适配
- IO 适配
- 点胶阀/供胶/加热/传感器接入
- 厂商 SDK 封装
- Fake device / mock device
- 统一设备状态与故障模型落地

### 不负责内容

- 工艺规则
- 机器运行模式规则
- 任务状态机
- HMI 展示逻辑

---

## 7.5 `packages/transport-gateway`

### 定位

承接旧仓中的 `tcp` 入口、协议封装、会话、消息编解码和 facade 映射逻辑。

### 负责内容

- TCP/JSON 协议绑定
- 请求/响应 envelope
- 会话管理
- 协议版本兼容策略
- 应用契约到传输消息的映射
- Python HMI 与 C++ 服务端的通信边界

### 不负责内容

- 业务规则实现
- HMI 页面逻辑
- 设备驱动

### 关键价值

原方案里缺少这一层，容易导致：

- HMI 直接和核心内部模型耦合
- TCP server 逻辑散落在入口应用中
- 协议兼容策略没有正式归属

---

## 7.6 `packages/traceability-observability`

### 定位

承接旧仓的日志、事件、记录、追溯、回放和查询模型能力，
并与核心领域中的 `diagnostics-domain` 配合。

### 负责内容

- 运行事件
- 报警记录
- 工艺结果留痕
- 参数快照
- 查询模型
- 回放数据结构
- 存储接口与实现抽象

### 不负责内容

- 核心控制闭环
- 工艺规则求值
- 安全互锁判定
- 设备驱动实现

### 说明

领域规则在 `process-runtime-core`，
记录/查询/回放/持久化在 `traceability-observability`。
这比把两者混成一个“诊断包”更稳定。

---

## 7.7 `packages/application-contracts`

### 定位

HMI、CLI、TCP facade 与核心应用服务之间的应用级契约。

### 内容形式

优先放：

- 命令/查询 schema
- DTO 定义
- 状态枚举
- 校验规则
- 事件模型
- codegen 输出
- 契约版本策略

### 特别要求

该包不能只放某一语言的内部实现类。
它必须能服务于：

- C++ 运行时
- Python HMI
- 未来脚本工具或自动化入口

---

## 7.8 `packages/device-contracts`

### 定位

`process-runtime-core` 与 `device-adapters` 之间的设备能力契约。

### 负责内容

- 设备命令模型
- 能力声明
- 状态模型
- 故障模型
- 轨迹执行模型
- 触发计划模型
- 设备事件模型

---

## 7.9 `packages/engineering-contracts`

### 定位

工程数据预处理输出的版本化契约层。

### 负责内容

- proto / schema
- JSON/PB 输出格式定义
- 兼容性测试
- 版本策略
- codegen

### 说明

旧仓中 DXF 到 `.pb` 的链路是真实存在的系统边界，
不能只把它埋在实现包里。

---

## 7.10 `packages/editor-contracts`

### 定位

HMI 与 DXF 编辑器之间的外部进程协作契约层。

### 负责内容

- 命令行参数约定
- notify 文件格式
- 编辑结果回传模型
- 错误码与状态约定

### 说明

旧系统中这条边界是稳定存在的。
如果重构中忽略它，后续会重新回到“靠约定俗成和路径拼接”的状态。

---

## 7.11 `packages/shared-kernel`

### 定位

小而稳的共享基础内核。

### 可放内容

- 通用 ID
- Result/Either
- 错误模型
- 时间/单位表达
- 小型数学工具
- 极少量稳定值对象

### 严禁内容

- 配方规则
- 工艺逻辑
- 机器状态机
- 任何“先放公共再说”的代码

---

## 7.12 `packages/simulation-engine`

### 定位

确定性时序仿真与软件在环支撑能力。

### 依赖要求

可依赖：

- `application-contracts`
- `device-contracts`
- `engineering-contracts`
- `shared-kernel`

不应被核心运行时反向依赖。

---

## 7.13 `packages/test-kit`

### 定位

跨包测试支撑与回归样本工具箱。

### 负责内容

- builders
- fixtures
- fake data
- golden files
- 契约兼容性断言
- 场景回归工具

## 8. 应用级方案

## 8.1 `apps/control-runtime`

运行时宿主入口。
用于装配核心能力、设备能力、配置、监控与后台流程。

## 8.2 `apps/control-tcp-server`

对外暴露 TCP/JSON 服务。
它是运行入口，不是核心业务包。

## 8.3 `apps/control-cli`

保留开发调试、无界面验证和自动化脚本入口。
CLI 在旧仓中是重要验证面，不建议消失。

## 8.4 `apps/hmi-app`

桌面 HMI 交互壳层。
只通过 `application-contracts` 和 `transport-gateway` 接入核心系统。

## 8.5 `apps/dxf-editor-app`

独立 DXF 编辑器。
它是稳定工具应用，不应被错误归入核心业务包或 `engineering-data`。

## 10. 与原 `重构.md` 的关键差异

修订版相对原方案的主要变化如下：

1. 补上了 `control-runtime`、`control-tcp-server`、`control-cli`、`dxf-editor-app`
2. 新增 `runtime-host`，承接旧仓真实存在的装配与宿主能力
3. 新增 `transport-gateway`，正式承接 TCP 协议和会话边界
4. 新增 `engineering-contracts` 和 `editor-contracts`
5. 明确 `process-runtime-core` 内部必须包含 `recipes / machine / safety / diagnostics-domain`
6. 明确跨语言契约应以 schema/codegen 为中心，而不是某端内部类
7. 明确 `data/` 是系统资产目录，而不是杂项目录

## 11. 推荐迁移顺序

## 阶段 0：做全量盘点，不急着移动目录

先完成三张表：

1. 运行入口清单
2. 契约清单
3. 领域能力清单

否则迁移只会继续沿着当前拆分结果走，无法真正对齐旧仓全貌。

## 阶段 1：先统一工作区治理，再保留子项目源码布局

先把并列子仓提升为真正统一工作区：

- 统一构建入口
- 统一测试入口
- 统一文档索引
- 统一 CI
- 统一依赖规则

此阶段不要急于大规模搬代码。

## 阶段 2：先确立契约层

优先把以下契约显式化：

- HMI <-> Core 应用契约
- Core <-> Device 设备契约
- DXF 预处理输出契约
- HMI <-> DXF Editor 协作契约

契约不稳时，物理拆包只会制造更多耦合。

## 阶段 3：重构应用入口

把入口整理为：

- `control-runtime`
- `control-tcp-server`
- `control-cli`
- `hmi-app`
- `dxf-editor-app`

先把“入口是什么”讲清楚，再去拆能力包。

## 阶段 4：收拢核心内核

将旧仓中的：

- `motion`
- `dispensing`
- `machine`
- `safety`
- `recipes`
- `planning`
- `trajectory`
- `diagnostics-domain`

统一收拢进 `process-runtime-core`，
但在包内设置依赖限制与子域边界。

## 阶段 5：剥离宿主与设备

把以下能力从核心中拉出：

- 容器装配 -> `runtime-host`
- 控制卡/硬件驱动 -> `device-adapters`
- TCP 协议与映射 -> `transport-gateway`

## 阶段 6：建立集成验证层

补齐：

- HMI 到 TCP 的契约兼容性测试
- DXF 到执行计划的场景回归
- 仿真回归
- 硬件冒烟

## 阶段 7：最后再考虑是否局部继续细拆

只有当以下条件成立时，才考虑把 `process-runtime-core` 再进一步物理拆包：

- 接口稳定
- 团队边界稳定
- 集成测试成熟
- 版本协同成本可控

否则继续保持“核心收拢、包内分层”更稳妥。

---

## 12. 最终结论

基于 `Backend_CPP` 的真实能力全貌，SiligenSuite 的目标形态不应只是：

**单仓多库 + hmi-app + core + device + contracts**

而应是：

**单仓多库 + 多应用入口 + 核心业务内核收拢 + 运行时宿主独立 + 跨语言契约显式化 + 工具应用独立化 + 集成验证一级化**

更准确地说：

```text
SiligenSuite 应建设为一个统一治理的 Monorepo。

其中 control-runtime、control-tcp-server、control-cli、hmi-app、dxf-editor-app 作为明确的应用入口；
process-runtime-core 作为强收拢业务内核，内部显式包含 process/planning/execution/machine/safety/recipes/dispensing/diagnostics-domain；
runtime-host 负责宿主装配与系统初始化；
device-adapters 负责硬件适配；
transport-gateway 负责 TCP/JSON 边界；
application-contracts、device-contracts、engineering-contracts、editor-contracts 负责跨边界协作；
simulation-engine、test-kit、integration 负责仿真、回归与联调验证。
```

这比原 `重构.md` 更完整地覆盖了旧仓能力，也更适合作为后续正式重构的基线方案。
