# SiligenSuite 单仓多库最终方案（定稿）

## 1. 最终结论

对于**无痕内衣点胶机上位机**，最终采用：

**单仓多库（Monorepo + Multi-Package）**

而不是：

- 按 7 个一级领域直接拆成 7 个独立仓库
- 完全单体、无明确边界的大仓混合开发

该方案的核心判断是：

1. 系统存在明确的业务闭环，但核心链路仍处于强协同演化阶段  
2. 直接多仓会过早固化尚未稳定的边界，放大接口、联调、版本协同成本  
3. 单仓多库既能建立清晰模块边界，又能保留原子变更、统一集成与统一发布能力  

---

## 2. 架构原则

本方案遵循以下原则：

### 2.1 以业务闭环为主，不以技术栈为主
系统核心链路是：

```text
设计输入 → 工艺定义 → 路径生成 → 任务执行 → 设备联动 → 结果追溯 → 人机交互
````

模块划分必须围绕这条闭环，而不是按“前端 / 后端 / 工具类”切。

### 2.2 一级领域不等于一级仓库

理论领域可以独立，但仓库边界还要考虑：

* 运行时耦合
* 联调频率
* 接口成熟度
* 发布节奏
* 测试方式
* 责任归属

因此，领域边界用于认知与治理，包边界用于工程落地。

### 2.3 核心链路先收拢，外围能力先独立

当前阶段最不适合继续拆细的，是：

```text
工艺规则 → 轨迹规划 → 任务执行
```

这三者应先收拢为一个核心包，在包内强模块化，而不是物理拆散成多个包或多个仓库。

### 2.4 契约必须显式存在

跨边界不允许依靠“内部对象顺手引用”完成协作，必须通过显式契约：

* 应用契约：HMI ↔ Core
* 设备契约：Core ↔ Device Adapters

### 2.5 集成测试是一级工程对象

工业设备软件不能只靠包内单元测试。
跨模块集成验证、仿真验证、硬件联调验证必须独立存在。

## 3. 最终模块方案

最终采用如下模块体系：

### 3.1 应用层

* `hmi-app`

### 3.2 核心包

* `engineering-data`
* `process-runtime-core`
* `device-adapters`
* `traceability-observability`

* `application-contracts`
* `device-contracts`

* `shared-kernel`

## 5. 顶层目录职责说明

### 5.1 `apps/`

放最终可运行应用。

当前仅保留：

* `hmi-app`

说明：

* HMI 是最终应用，不是公共能力包
* 它可以依赖契约包与核心服务，但不允许被任何核心包反向依赖

### 5.2 `packages/`

放可复用、可测试、可受控依赖的能力包。
这是整个 monorepo 的核心区域。

### 5.3 `integration/`

放跨包协同验证、场景测试、仿真联调、硬件联调与回归基线。
不能把所有测试都塞到各包内部。

### 5.4 `tools/`

放构建、代码生成、静态检查、发布与辅助脚本。

### 5.5 `docs/`

放架构文档、领域文档、契约文档、决策记录与入门文档。
关键边界必须文档化，不允许只存在于口头理解中。

### 5.6 `config/`

放测试、构建、环境、CI 等统一配置。

### 5.7 `examples/`

放 DXF 示例、工艺样例、任务计划样例、回放样例。
用于测试、演示与回归。

#### 关键说明

当前阶段**不建议**把该包继续拆成：

* 工艺包
* 轨迹包
* 执行包

#### 核心边界原则

* `process` 只定义做什么，不直接持有设备补丁逻辑
* `planning` 只负责形成可执行轨迹，不直接绑定具体控制卡
* `execution` 只负责如何运行计划，不直接持有硬件专有实现
* `application` 只对外提供服务，不应越级访问内部私有实现

真正共享且足够稳定的基础公共内核。

* Result/Either 类型

* 时间表达
* 单位表达
* 数学基础工具

#### 严禁内容

* 任何边界不清的业务逻辑
* “因为不知道放哪就先放公共”的代码

#### 建议内部结构

```text
shared-kernel/
├── src/
│   ├── primitives/
│   ├── ids/
│   ├── results/
```

**shared-kernel 必须小而稳，不允许演化为公共垃圾堆。**

## 11. 最终工程治理规则

### 11.1 每个包必须有 README

至少写明：

* 负责什么
* 不负责什么
* 对外提供什么
* 依赖什么
* 哪些依赖被禁止

### 11.2 每个包必须具备独立测试能力

不能存在“只有生产代码，没有本地验证能力”的包。

### 11.3 契约包必须保持纯净

* 不放实现
* 不放副作用逻辑
* 不依赖实现包

### 11.4 `shared-kernel` 必须设边界审查

超过约定体量后应审查，防止其膨胀。

### 11.5 `process-runtime-core` 内部也要执行依赖限制

例如：

* `execution` 不应直接操纵 `process` 私有规则实现
* `application` 不应越级访问 `planning` 私有实现
* `planning` 不应绑定具体控制卡语义

### 11.6 集成验证必须持续存在

不能因包内单测完善就取消场景回归、仿真回归与硬件冒烟验证。

---

## 12. 方案收益与取舍结论

### 12.1 本方案的主要收益

* 在不引入多仓高协同成本的前提下建立清晰边界
* 保留跨模块原子变更能力
* 保留统一构建、统一测试、统一发布能力
* 让外围稳定能力先独立治理
* 让核心链路先在包内稳定，而非过早物理拆散
* 为未来“局部多仓化”保留演进空间

### 12.2 本方案的主要取舍

* 不追求一开始把所有领域物理隔离
* 不追求形式上的“每个领域一个包”
* 明确接受：`process-runtime-core` 在当前阶段是强收拢内核
* 用显式契约和依赖规则来控制边界，而不是依赖目录命名自我约束

---

## 13. 最终判断

该系统最终采用：

**单仓多库 + 核心链路收拢 + 契约显式化 + 集成验证独立化**

这是当前阶段最符合无痕内衣点胶机上位机工程现实的方案。

精确表达如下：

```text
SiligenSuite 当前应建设为一个以 Monorepo 为载体的多包工程，
其中 engineering-data、device-adapters、traceability-observability、hmi-app 形成清晰边界，
process-runtime-core 作为核心业务内核先收拢治理，
并通过 application-contracts 与 device-contracts 管控跨边界协作。
```
