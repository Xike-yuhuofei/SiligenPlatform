# SS Incident Debug

## 适用边界

当用户希望你先定位原因，而不是马上改代码时，使用本技能。典型触发包括：

1. 先分析
2. 先排查
3. 先定位原因
4. 先审查链路
5. 评估修复落点
6. 审查模块职责、边界或契约是否漂移

本技能用于“问题定义 + 根因定位 + 证据收敛”。

下列场景不要硬套本技能：

1. 用户要求批量做无机台回归，改用 `ss-mock-regression`
2. 用户要求深挖单个功能的 mock/HMI/协议测试链路，改用 `ss-feature-deep-test`
3. 用户已经明确要求直接实现代码修复，先完成最小必要分析后转入正常实现流程

## 强制约束

1. 全程使用简体中文，保持严格中立、专业、工程导向。
2. 开始前先执行 `scripts/validation/resolve-workflow-context.ps1`。
3. 若确实需要高风险命令，改走 `scripts/validation/invoke-guarded-command.ps1 -Command "<your command>"`。
4. 先定义问题，不要把用户的模糊表述直接当成已定义问题。
5. 先建立证据链，不要把猜测写成事实。
6. 先定位异常首次出现层，再讨论修复。
7. 至少给出 3 个候选根因假设，不要只报一个“我觉得大概率”的结论。
8. 证据不足时必须明确写出“证据不足”。
9. 如果用户要求“先定位原因”或“先分析”，停止在分析与证伪设计阶段，不输出补丁、伪实现或直接改代码建议。

## 三个强制思维框架

### 1. 苏格拉底式提问

持续追问，直到问题被收敛为可验证陈述：

1. 现象到底是什么
2. 期望行为到底是什么
3. 异常首次出现在哪一层
4. 当前证据是什么
5. 哪些前提只是猜测，还未验证
6. 当前表达的是现象、推测还是结论
7. 问题属于几何、工艺、运动、执行、通信、状态机、HMI、视觉、机械、材料中的哪一类
8. 当前看到的是根因，还是下游表现

### 2. 第一性原理

不要只围绕某个函数或模块名思考，强制把问题映射到主链路：

1. 输入几何
2. 几何归一化
3. 工艺路径
4. 运动规划
5. 执行段生成
6. IO 时序
7. 设备执行 / 反馈
8. HMI 展示 / 追溯

对每一层都回答：

1. 输入是什么
2. 输出是什么
3. 不变量是什么
4. 允许变化项是什么
5. 这一层如果出错，会如何体现在下游

### 3. 奥卡姆剃须刀

对候选根因和候选修复方案排序时，优先考虑：

1. 是否位于异常首次出现层
2. 是否能解释最多关键现象
3. 是否依赖更少额外假设
4. 是否最近有代码、配置或契约改动
5. 是否能快速证伪
6. 是否改动更局部、边界更清晰、影响面更可控

## 执行流程

### 1. 问题本质重述

先用工程语言重述你理解到的问题本质，不要简单复述用户原话。至少说明：

1. 这是哪个层级的问题
2. 影响范围是什么
3. 当前更像功能不符合预期、链路契约失真，还是执行结果异常

### 2. 现象与期望

明确写出：

1. 实际现象
2. 期望行为
3. 判断依据
4. 复现条件
5. 影响范围
6. 当前证据
7. 当前尚未验证的假设

如果这一部分不清晰，不要进入后续方案设计。

### 3. 主链路拆解

按以下顺序逐层拆解：

1. 输入几何
2. 几何归一化
3. 工艺路径
4. 运动规划
5. 执行段生成
6. IO 时序
7. 设备执行 / 反馈
8. HMI 展示 / 追溯

每层至少记录：

1. 输入对象
2. 输出对象
3. 不变量
4. 允许变化项
5. 本层异常的下游表现

### 4. 候选根因假设

至少给出 3 个候选假设。每个假设必须包含：

1. 假设内容
2. 为什么能解释当前现象
3. 当前支持证据
4. 当前反对证据
5. 缺失证据
6. 如何证伪
7. 当前优先级

### 5. 假设排序与依据

必须明确说明：

1. 为什么优先查 A
2. 为什么 B 次之
3. 为什么 C 暂不优先

只列编号不算完成排序。

### 6. 最小证伪动作

先给最小证伪动作，再谈代码修改。每个动作尽量满足：

1. 改动少或不改代码
2. 采集证据直接
3. 能排除高优先级假设
4. 对现有系统侵入性低

每个动作按以下字段给出：

1. 目标假设
2. 观测点
3. 需要采集的数据
4. 预期结果
5. 判定标准

### 7. 方案评估

仅当用户明确要求继续进入方案阶段，或者证据已经足够时，才进入修复方案评估。

每个方案必须说明：

1. 改动落在哪一层
2. 为什么这一层是正确落点
3. 为什么不是上游或下游责任
4. 优点
5. 风险
6. 契约影响
7. 回归影响
8. 是否引入新增概念、新抽象或新耦合
9. 是否符合奥卡姆剃须刀

## 工业点胶场景专用判断

### DXF

默认认知：

1. DXF 提供的是几何意图
2. DXF 不是工艺语义
3. DXF 不是最终执行轨迹
4. DXF 不是板卡程序

不要从“DXF 看起来对”直接推出“执行一定对”。

### 预览

默认认知：

1. 预览正确不等于执行正确
2. 预览异常不等于底层执行一定异常
3. 必须先确认预览绑定的数据源来自哪一层
4. 必须区分几何预览、工艺预览、轨迹预览、执行预览、胶点预览

### 点胶质量

不要把点胶质量简化成单一空间点位问题。至少同时考虑：

1. 空间路径
2. 时间规律
3. 速度变化
4. 加减速策略
5. 阀开闭时序
6. 阀响应延迟
7. 供胶条件
8. 材料特性
9. 设备姿态与标定状态

### 拐角、首尾、局部异常

优先考虑：

1. 曲率变化
2. 角点减速
3. 首尾补偿
4. 出胶启停延迟
5. 路径段拼接
6. 轨迹重采样
7. 执行段裁剪
8. 预览绑定错误

没有直接证据前，不要先归因于材料不稳定或机械误差。

### 在线 / 离线模式

默认认知：

1. 离线正常，只能说明部分上游链路正常
2. 离线正常不能证明 TCP、硬件握手、板卡执行、IO 时序、在线 supervisor 状态门控正确
3. 在线异常时，优先检查 backend ready、tcp ready、hardware ready、online ready 门控、执行前置条件检查

## 输出要求

默认输出按以下结构组织：

1. 问题本质
2. 现象与期望
3. 当前证据与未验证前提
4. 主链路拆解
5. 候选根因假设
6. 假设排序与依据
7. 下一步最小证伪动作
8. 方案评估与推荐（仅证据足够时）
9. 回归检查点
10. 证据不足项

如果用户要“可执行排查清单”，优先按以下字段输出：

1. 文件路径 / 模块
2. 当前责任
3. 应验证的事实
4. 预期证据
5. 异常信号
6. 若失败说明什么
7. 关联的候选假设

如果用户只要求“先定位原因”，只输出分析与证伪设计，不要输出代码补丁。
