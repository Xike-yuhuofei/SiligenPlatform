# SS Mock Regression

## 适用场景

当用户希望你：

1. 不接真实机台、不连真实硬件
2. 对一组功能做批量回归测试
3. 优先复用仓库内已有测试资产
4. 不只给方案，而是直接执行并给出结论

本技能用于“批量功能回归”，不是单功能深挖；单功能深挖应改用 `ss-feature-deep-test`。

## 强制约束

1. 全程使用简体中文。
2. 不使用真实机台，不连接真实硬件。
3. 优先复用现有：
   - mock server
   - smoke 脚本
   - 协议层联调入口
   - 单元测试
   - 集成测试
   - ctest / 聚合测试二进制
4. 不要停留在分析；默认直接执行。
5. 不要因为第一次命令失败就停止；先定位、修复路径、补构建产物、切换正确入口，再继续执行。
6. 不要默认全量重编；如需构建，只构建最小必要目标。
7. 先不要改业务代码；只有在现有覆盖明显不足时，才允许补最小必要测试。

## 项目内执行约束

1. 开始前先执行：

```powershell
scripts/validation/resolve-workflow-context.ps1
```

2. 如果确实需要执行高风险命令，改走：

```powershell
scripts/validation/invoke-guarded-command.ps1 -Command "<your command>"
```

对普通只读命令、测试命令、最小构建命令，不要滥用 guarded wrapper。

## 执行流程

### 1. 锁定测试范围

从用户请求中提取：

1. 功能列表
2. 是否明确要求无机台
3. 是否要求优先复用现有测试资产
4. 是否允许补最小测试

如果用户没写功能列表但要求“批量扫一遍”，先扫描仓库中适合无机台验证的功能，再向下执行。

### 2. 盘点测试入口

优先查找：

1. HMI / UI smoke 入口
2. mock server
3. 协议 smoke 或工具脚本
4. 单元测试文件
5. 聚合测试二进制与 `ctest` 入口
6. 功能对应的领域层/应用层实现

搜索优先使用 `rg`。

### 3. 按三层顺序验证

每个功能默认按以下顺序推进：

1. HMI / UI smoke
2. 协议层 + mock 联调
3. 领域层 / 规则层测试

如果某一层不存在对应入口，要明确记录“该层无现成入口”，再继续下一层。

### 4. 每项功能至少覆盖的内容

尽量覆盖：

1. 正常路径
2. 前置条件检查
3. 互锁拦截
4. stop / release / close / pause / resume 等收尾动作

如果某项能力与另一功能共用同一条链路，要说明：

1. 共用了哪一层
2. 额外验证了哪部分差异

### 5. 处理测试入口异常

如果出现以下情况，不要直接结束：

1. `ctest` 找不到测试
2. 测试存在但当前工作区没有构建产物
3. 测试在聚合二进制里而不是独立目标
4. 路径错误或构建配置不匹配

需要继续做：

1. 定位真实测试目标
2. 定位实际二进制路径
3. 补构建最小目标
4. 用正确的过滤方式执行相关用例

### 6. 需要补测试时的边界

只有在现有资产不足以覆盖用户明确要求的功能时，才补最小必要测试。

补测试时遵守：

1. 先补测试，不先改业务代码
2. 只补当前功能最小覆盖面
3. 优先补 mock / 协议 / 规则层测试
4. 补完必须实际执行

## 输出要求

过程更新要简短，说明：

1. 正在测什么
2. 用了什么入口
3. 遇到什么问题
4. 如何继续推进

最终报告必须包含：

1. 已通过
2. 失败或异常
3. 未覆盖项
4. 汇总表

汇总表至少包含：

1. 功能
2. 测试层级
3. 是否执行
4. 是否通过
5. 证据
6. 风险/未覆盖项

## 结果边界

必须明确区分：

1. “mock / 无机台验证通过”
2. “真实机台现场通过”

禁止把 mock 通过直接等同于真实上线通过。

通常仍未覆盖的内容包括：

1. 电机真实运动方向
2. 真实 IO 接线
3. 伺服响应
4. 真实限位 / 急停硬件链路
5. 停止距离、抖动、过冲、丢步等真实运动行为
