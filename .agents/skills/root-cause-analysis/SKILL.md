# Purpose

本 skill 的唯一职责是：对故障修复流中的问题做根因定位，而不是直接修代码。

它的目标不是“猜一个看起来像的原因”，而是：

- 明确主假设与次假设
- 给出支持证据与反证
- 给出最小证伪动作
- 收敛到当前最可能根因
- 明确哪些地方仍证据不足

# Use this skill when

- 已经完成故障定义与初步边界。
- 已经拿到了日志、报错、复现路径、样本或代码线索。
- 当前目标是先定位原因，再做最小修复。
- 用户明确要求“先定位原因”“先找根因”。

# Do not use this skill when

- 当前是新功能设计。
- 当前已经确定根因，只差执行修复。
- 当前缺少最基本的故障定义、复现路径或证据材料。
- 当前只是做收尾或文档同步。

# Inputs

- 故障定义
- 影响范围
- 复现路径
- 日志、报错、截图、轨迹样本
- 相关链路代码
- 契约冻结结果（若适用）
- 用户给出的额外观察或排除项

# Required outputs

1. 主假设
2. 次假设
3. 支持证据
4. 反证 / 弱证
5. 最小证伪动作
6. 当前最可能根因
7. 后续建议

# Hard prohibitions

- 不在证据不足时给绝对结论。
- 不直接夹带修复代码。
- 不只给一个孤立猜测。
- 不跳过复现和链路对照。
- 不把现象描述直接当根因。

# Analysis rules

## 1. 先区分现象、触发条件、根因

- 现象：用户看到什么
- 触发条件：在什么输入 / 环境下出现
- 根因：为什么会出现

三者不能混写。

## 2. 根因必须沿链路收敛

优先沿以下链路找断点：

- 输入是否合法
- 语义是否被正确解释
- 中间工件是否丢失信息
- 适配层是否改写语义
- 展示层是否依赖了错误来源
- 错误是否因状态前提不满足而触发

## 3. 必须至少给出两个可竞争假设

如果只给一个假设，说明分析还不充分。

## 4. 最小证伪优先于大范围重构

优先建议补观测点、对照样本、最小单测、最小日志和单链路隔离，而不是先整体重构或先改多个模块再看结果。
