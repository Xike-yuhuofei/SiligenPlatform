# Xike-CodeExplain

## Overview

将任意代码目录整理成两段式文档：

- 文件树：只展示结构
- 表格：解释实现文件在业务主线中的作用

输出重点不是“这个文件是什么类”，而是“这个文件在业务流程里起什么作用”。

## Workflow

### 1. 锁定输入

先确认这 3 个输入：

| 输入项 | 说明 |
|---|---|
| `目标目录` | 本次要整理的目录 |
| `业务主线` | 所有说明围绕的业务流程 |
| `结果文件` | 最终输出文档路径 |

如果用户额外指定纳入或排除规则，以用户规则覆盖默认规则。

### 2. 生成文件树

文件树只保留结构，不附任何说明。

默认纳入：

- `.cpp`、`.cc`、`.cxx`
- `.h`、`.hpp`
- `.py`
- `.json`、`.yaml`、`.yml`、`.toml`
- `CMakeLists.txt`
- 元数据文件，例如 `module.yaml`

默认排除：

- `.md`
- `.gitkeep`
- `.pyc`
- `.txt`
- `.def`
- `.exp`

### 3. 划分表格

按最后一级文件夹分组。

- 找出所有包含实现文件的叶子目录
- 每个叶子目录对应一张表
- 表格默认只纳入实现文件

表格默认纳入：

- `.cpp`
- `.cc`
- `.cxx`
- `.py`

表格默认排除：

- `CMakeLists.txt`
- 元数据文件，例如 `module.yaml`
- 所有 `.h/.hpp`
- 文档、缓存、占位文件

### 4. 填写业务表格

表格固定使用 5 列：

| 列名 | 要求 |
|---|---|
| `流程阶段` | 写业务阶段，不写技术分层 |
| `触发条件` | 写清楚何时进入这段逻辑 |
| `当前职责` | 写清楚这个文件直接做什么 |
| `输出结果` | 写清楚结果交给谁或给下一段做什么 |
| `失败影响` | 写清楚出错后的现场表现或链路后果 |

每一行都必须回答这 5 个问题：

- 位于哪一段
- 什么时候触发
- 当前做什么
- 输出什么
- 失败会怎样

说明必须围绕业务主线。比如业务主线是“点胶流程”，就要写它在点胶前准备、校准、路径生成、执行、收尾里的作用，而不是写“这是某某模块的实现文件”。

### 5. 复查质量

逐行检查：

- 有没有明确的流程阶段
- 有没有具体的触发条件
- 有没有直接动作，而不是抽象表述
- 有没有下游输出对象或下一环节
- 有没有失败后果

以下写法判定不合格：

- “这是某某类的实现文件”
- “负责某某逻辑”
- “用于处理相关业务”
- “位于某某目录，承担某某能力”

如果缺少流程位置、触发条件、输出结果或失败影响，直接重写。

## Output Contract

结果文档固定分两部分：

1. 文件树
2. 表格

文件树展示结构；表格展示业务解读。不要在文件树里追加解释。

## Templates

### 结果文档头部模板

```md
# <目标目录名称> 文件树与业务解读

本文档分两部分：

- 文件树：展示 `<目标目录>` 的代码结构，不附文件说明。
- 表格：按最后一级文件夹展示实现文件在 `<业务主线>` 中的具体作用。

- 目标目录：`<目标目录>`
- 业务主线：`<业务主线>`
- 表格列：`流程阶段`、`触发条件`、`当前职责`、`输出结果`、`失败影响`
- 表格默认纳入：`.cpp`、`.cc`、`.cxx`、`.py`
- 表格默认排除：`CMakeLists.txt`、元数据文件、`.h/.hpp`、文档、缓存、占位文件
```

### 文件树模板

````md
## <分组或模块名>

**文件树**
```text
<目标目录或模块名>/
├── <子目录>/
│   ├── <文件A>
│   └── <文件B>
└── <文件C>
```
````

### 表格模板

```md
**<叶子目录路径>/**

| 文件 | 流程阶段 | 触发条件 | 当前职责 | 输出结果 | 失败影响 |
|---|---|---|---|---|---|
| `<文件名>` | `<位于哪一段>` | `<什么时候进入>` | `<直接做什么>` | `<输出给谁/下一段做什么>` | `<出错后会怎样>` |
```

### 单文件草稿模板

```text
流程阶段：位于 <业务流程阶段>。
触发条件：当 <触发条件> 时进入。
当前职责：负责 <当前动作>。
输出结果：输出 <输出对象>，交给 <下游对象/下游环节>。
失败影响：如果这里出错，会导致 <现场表现/链路后果>。
```
