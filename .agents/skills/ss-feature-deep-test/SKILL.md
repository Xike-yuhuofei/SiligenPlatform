# SS Feature Deep Test

## 适用场景

当用户希望你：

1. 只聚焦一个功能
2. 沿着 HMI / 协议 / 领域三层打透
3. 找到真实测试入口而不是停在表面 smoke
4. 必要时继续补构建、过滤聚合测试、补最小测试

本技能用于“单功能深挖”，不是批量回归；批量回归应改用 `ss-mock-regression`。

## 强制约束

1. 全程使用简体中文。
2. 不使用真实机台，不连接真实硬件。
3. 默认直接执行，不只给方案。
4. 优先复用仓库已有 mock、smoke、unit test、ctest、协议测试入口。
5. 不要默认全量构建；只构建与该功能相关的最小必要目标。
6. 先不要改业务代码；只有覆盖明显不足时，才补最小必要测试。

## 项目内执行约束

1. 开始前先执行：

```powershell
scripts/validation/resolve-workflow-context.ps1
```

2. 如果确实需要执行高风险命令，改走：

```powershell
scripts/validation/invoke-guarded-command.ps1 -Command "<your command>"
```

## 执行流程

### 1. 锁定单一功能

明确本轮只测一个功能，例如：

1. 点动
2. 回零
3. 点胶阀控制
4. 供胶控制
5. 急停
6. move / move_to

如果用户一次给了多个功能，不要用本技能硬做批量回归；应切换到 `ss-mock-regression`。

### 2. 建立三层定位图

围绕该功能定位：

1. HMI 入口
2. 协议方法 / transport 命令
3. 应用层用例
4. 领域层核心规则
5. mock 行为
6. 单元测试 / 聚合测试 / ctest 入口

目标不是只找到文件，而是弄清：

1. 请求从哪里发起
2. 经过哪条协议链路
3. 最终由哪段规则控制

### 3. 按层执行

优先顺序：

1. 最接近真实链路的 smoke 或 HMI 行为
2. 协议层 + mock 联调
3. 领域层 / 规则层测试

如果某层没有现成入口，明确记录，再继续下一层。

### 4. 深挖时必须覆盖的内容

至少覆盖：

1. 正常路径
2. 前置条件检查
3. 互锁拦截
4. 停止 / 关闭 / 释放 / 收尾动作
5. 三层行为是否一致

### 5. 处理聚合测试与构建产物缺失

如果遇到：

1. 测试源码存在，但 `ctest` 名称不直观
2. 测试被编进聚合二进制
3. 当前工作区没有对应测试产物

必须继续做：

1. 追到真实构建目标
2. 定位真实二进制
3. 最小构建
4. 用 `gtest_filter` 或等价方式执行目标用例

不能只说“测试存在”却不执行。

### 6. 需要补测试时的边界

只有在以下条件成立时才补测试：

1. 用户明确要求该功能必须验证
2. 仓库内现成资产无法覆盖关键路径

补测试时优先：

1. mock 行为测试
2. 协议契约测试
3. 领域规则测试

补完必须实际执行。

## 输出要求

过程更新需要持续说明：

1. 正在测哪一层
2. 用了什么入口
3. 当前发现了什么
4. 下一步要补什么

最终输出必须包含：

1. 使用了哪些命令和入口
2. 实际执行结果
3. 明确结论：通过 / 失败 / 阻塞
4. 当前无机台测试覆盖了什么
5. 当前无机台测试没有覆盖什么

## 结论边界

必须明确区分：

1. 功能的软件链路通过
2. 功能的真实硬件行为通过

无机台深挖通常不能证明：

1. 真实驱动器 / IO / 接线
2. 真实运动方向
3. 真实停机距离、抖动、过冲、丢步
4. 真实限位 / 急停硬件闭环
