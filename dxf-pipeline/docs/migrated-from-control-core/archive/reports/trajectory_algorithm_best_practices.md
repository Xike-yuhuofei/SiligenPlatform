## 1. 背景
为了提升 Siligen 点胶控制系统的点胶质量（均匀性、无堆胶）和运行效率，需要引入行业最佳的轨迹规划算法。目前系统支持基础的 DXF 解析和线性插补，但在拐角处理和速度连续性方面存在优化空间。

## 2. 行业最佳实践 (调研结果)

通过对工业点胶领域的调研，总结出以下关键技术：

### 2.1 拐角平滑 (Corner Smoothing / Blending)
*   **问题**: 机械结构无法瞬时改变速度方向。未经处理的直线连接（尖角）迫使机器减速至零再加速，导致点胶堆积（"狗骨头"效应）和振动。
*   **解决方案**: 在直线交接处插入过渡曲线。
    *   **圆弧过渡**: 最简单，计算量小。
    *   **B-Spline/Bezier**: 更高阶的平滑（G2连续），适合高速高精场景。
*   **效益**: 允许机器以非零速度通过拐角，保持胶宽一致。

### 2.2 速度前瞻 (Velocity Look-ahead)
*   **问题**: 仅看当前段无法预知前方急转弯，可能导致冲出轨迹或急刹车。
*   **解决方案**:
    *   **反向扫描**: 根据前方路径的几何约束（曲率），反推当前允许的最大速度。
    *   **S形加减速**: 避免梯形速度曲线带来的加加速度（Jerk）冲击。
*   **效益**: 充分利用机器加减速能力，缩短周期时间。

### 2.3 流量-速度同步 (Flow-Velocity Synchronization)
*   **最佳实践**: 点胶阀的开关/PWM控制应与运动路径的实时速度严格同步。对于变速度段（如拐角），需调整流量或接受微小的线宽变化（通常通过保持恒定路径速度来解决）。

## 3. 当前系统现状分析

### 代码库分析
*   **文件**:
    *   `src/infrastructure/adapters/planning/dxf/internal/DXFTrajectoryGenerator.cpp`
    *   `src/domain/motion/domain-services/interpolation/LinearInterpolator.cpp`
*   **现状**:
    *   **DXFTrajectoryGenerator**: 采用逐段独立插补。每条直线/圆弧被视为独立运动，连接处速度通常归零（或未显式处理连续性）。
    *   **LinearInterpolator**: 已包含 `LookAheadSpeedOptimization` 和 `SShapeLinearInterpolation` 的基础实现，但在 DXF 导入流程中未被有效利用。
    *   **缺失**: 缺乏几何层面的“拐角过渡”生成逻辑；缺乏全局的速度规划流程。

## 4. 候选方案详解

### 4.1 方案 A: 增强型原生实现 (几何平滑 + 速度前瞻)

本方案旨在在现有架构基础上，集成工业级的平滑算法。

**核心逻辑**:
1.  **几何预处理 (Geometry Pass)**: 将离散 DXF Segment 链接为 Path；在拐角处插入过渡圆弧（Blending Arc）替换尖角。
2.  **速度规划 (Velocity Pass)**: 计算每段（直线、过渡圆弧）的物理限制速度；执行双向扫描（反向确保减速足够，正向确保加速合规）。
3.  **插补生成 (Interpolation)**: 基于规划好的初末速度执行 S-Curve 插补。

**架构设计**:
*   新增 `GeometryBlender`: 负责几何修剪和圆弧插入。
*   新增 `TrajectoryPlanner`: 协调几何和速度。
*   复用 `LinearInterpolator`: 作为底层插补核心。

**实施计划**:
1.  **重构**: 修改 `DXFTrajectoryGenerator` 输出 `std::vector<MotionSegment>` 中间格式。
2.  **几何**: 实现直线-直线、直线-圆弧的圆弧过渡计算。
3.  **速度**: 实现全局双向速度扫描算法。
4.  **验证**: 可视化与实机测试。

**利弊**:
*   ✅ **优点**: 平滑度高（G1/G2连续），运行效率高（拐角不停车），代码自主可控。
*   ❌ **缺点**: 几何算法开发难度中等，需处理极短线段等边界情况。

### 4.2 方案 B: 简易速度前瞻 (仅速度优化)

本方案仅引入全局速度规划，不修改几何路径（保留尖角）。

**核心逻辑**:
1.  **路径链接**: 将 DXF Segment 链接为 Path。
2.  **速度规划**: 强制设定拐角处的通过速度为 0（或极低值，取决于电机允许的瞬时速度突变）。
3.  **前瞻优化**: 利用双向扫描算法，确保在到达尖角前提前减速，避免急刹车。

**架构设计**:
*   仅需增强 `DXFTrajectoryGenerator` 和 `LinearInterpolator` 的速度规划部分。
*   无需 `GeometryBlender`。

**实施计划**:
1.  **重构**: 修改 `DXFTrajectoryGenerator` 输出 `std::vector<MotionSegment>`。
2.  **速度**: 实现全局双向速度扫描算法（拐角速度设为0）。
3.  **验证**: 测试减速效果。

**利弊**:
*   ✅ **优点**: 开发成本低，复用现有逻辑多，风险最小。
*   ❌ **缺点**: 尖角处机器必须减速至接近停止，**生产效率低**，无法解决堆胶问题（因为速度降为0时胶阀响应可能滞后）。

### 4.3 方案 C: 集成第三方开源库

引入成熟的开源运动控制库来接管轨迹规划。

**候选库**:
*   **Ruckig**: 实时在线轨迹生成 (OTG)。
*   **Robotics Library (RL)**: 全功能机器人库。
*   **Open Motion Planning Library (OMPL)**: 路径规划。

**核心逻辑**:
1.  **适配层**: 编写 Adapter 将 DXF 数据转换为第三方库的输入格式。
2.  **调用**: 调用第三方库生成带时间戳的轨迹点。
3.  **转换**: 将结果转换为 Siligen 的 `TrajectoryResult`。

**实施计划**:
1.  **选型**: 评估并选择合适的库（推荐 Ruckig 用于 OTG）。
2.  **集成**: 配置 CMake/Vcpkg 引入依赖。
3.  **适配**: 编写数据转换层。
4.  **验证**: 对比原生实现。

**利弊**:
*   ✅ **优点**: 算法极其成熟，性能通常优于手写，支持更多高级特性（如Jerk限制）。
*   ❌ **缺点**: **集成成本高**（构建系统、依赖管理）；可能引入不必要的功能冗余；学习曲线陡峭；部分库（如 LinuxCNC）难以剥离；协议风险（GPL vs MIT）。
