# SiligenSuite 阶段0 迁移映射

## 1. 判定前提

本映射基于以下四类真实输入共同整理，而不是只复述目标方案：

- `D:\Projects\SiligenSuite\整体重构方案（修订版）.md`
- `D:\Projects\SiligenSuite\阶段0-基线盘点.md`
- 当前工作区 `D:\Projects\SiligenSuite` 的真实目录与源码
- 旧仓 `D:\Projects\Backend_CPP` 的真实目录与源码

阶段 0 的结论边界如下：

- 本文最初整理时，当前工作区根目录还没有真正落地顶层 `apps/`、`packages/`、`integration/` 等统一 monorepo 结构；其中仿真工程当时仍以顶层 `simulation-engine` 目录存在。
- `control-core/build/*` 中出现的目标不能当作源码事实；例如 `control-cli` 目前只有构建产物痕迹，没有对应的现存新式源码目录。
- 本文件只给出最终可执行的迁移归宿、兼容策略与阻塞项，不直接移动大规模代码。

| HMI 仍以 sibling repo 假设启动 backend | 统一工作区后的运行方式、部署方式 | `BackendManager` 默认指向同级 `control-core`，并直接扫描 `build/bin` | 在 monorepo 中提供稳定 app 路径或统一 launcher，替换 repo 相对路径发现逻辑 |

## 6. 可执行结论

当前最稳妥的执行口径不是“立刻搬目录”，而是按以下顺序推进：

1. 先把四条关键契约独立标出 owner：`application-contracts`、`device-contracts`、`engineering-contracts`、`editor-contracts`。
2. 再补齐缺失入口 `apps/control-cli`，并确认 `control-runtime`、`control-tcp-server` 的新路径为唯一 canonical 入口。
3. 然后把 `control-core` 里的现有种子模块重新对齐到最终包名：`process-core`/`motion-core` 汇总为 `process-runtime-core`，`device-hal` 对齐 `device-adapters`，`control-gateway` 对齐 `transport-gateway`。
4. 同步把 `control-core/data/recipes`、`config/machine_config.ini`、`dxf-pipeline/proto` 这些资产级目录提升到未来顶层 `data/`、`config/`、contracts 包的明确 owner 下。
5. 最后再拆除 `src/*` legacy 兼容壳和 sibling-repo 启动假设。

在以上动作完成之前，以下内容必须继续显式标记为“未最终落位”：

- `apps/control-cli`
- `packages/traceability-observability`
- `packages/application-contracts`
- `packages/device-contracts`
- `packages/engineering-contracts`
- `packages/editor-contracts`
- `Backend_CPP/uploads/*` 的运行时目录策略
- `Backend_CPP/tools/trajectory_compare` 的最终承接位置
