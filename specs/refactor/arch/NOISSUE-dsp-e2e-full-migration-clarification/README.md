# 本次任务理解冻结说明

更新时间：`2026-03-25`

## 1. 定位

本目录专用于承载“以 `docs/architecture/dsp-e2e-spec/` 为唯一正式冻结标准，完成整仓新架构现实收敛并物理清除旧架构资产”的任务理解说明。

本文件是当前轮次的任务理解冻结稿，不承担实施方案、技术选型、任务拆解或重构建议职责。

## 2. 任务本质

本次任务不是从零设计一套新架构，也不是继续允许新旧架构长期并行。

本次任务的本质是：

1. 以 `docs/architecture/dsp-e2e-spec/` 作为仓库唯一正式冻结事实源。
2. 面向当前仓库已经存在的代码、构建、测试、契约和样本资产，完成整仓级重新归位与现实收敛。
3. 让新架构目录从“统一骨架或承载壳”变成“唯一真实 owner 面”。
4. 在真实实现完成迁入后，物理清除所有 legacy 承载层与兼容残留。

## 3. 当前仓库现实

当前仓库已经完成部分迁移，但尚未完成最终收敛。

已确认的现实如下：

- `docs/architecture/dsp-e2e-spec/` 已经是仓库唯一正式冻结文档集。
- `modules/` 已经采用 `dsp-e2e-spec` 建议的统一骨架。
- 顶层 `packages/` 已经不存在，但旧架构并未彻底消失。
- 大部分真实实现、真实构建关系或兼容承载仍以旧架构语义保留在仓库中。
- legacy 不仅表现为旧顶层目录，也表现为嵌入新目录内部的旧架构子树、bridge、forwarder、wrapper、兼容 `CMakeLists.txt` 和旧语义组织方式。

因此，当前问题不是“是否已经有新架构目录”，而是“真实代码和真实构建是否已经完全落到新架构规定的位置”。

## 4. 范围理解

本次任务按整仓级口径成立，不区分“主链严格、非主链放宽”的双重完成标准。

范围内对象包括：

- 所有模块的真实源码承载位置
- 所有模块的真实构建定义与依赖入口
- 与模块 owner 直接相关的测试、契约、样本和配套资产
- 与新旧架构切换直接相关的目录承载关系和 legacy 残留

本次任务要求所有模块都达到同等级别的现实收敛。

## 5. 完成定义

当且仅当以下条件同时成立时，才能认为本次任务完成：

1. 所有模块的真实实现已经整理并分布到新架构对应目录下。
2. 新架构目录成为唯一真实实现面、唯一真实构建面和唯一真实 owner 面。
3. 仓库内不再保留旧架构承载层，不再保留 bridge、forwarder、wrapper、tombstone 或其他兼容壳作为终态组成部分。
4. legacy 资产被物理清除，而不是只降级为“非正式但仍保留”。
5. 仓库现实与 `dsp-e2e-spec` 在模块边界、生命周期、状态语义、接口契约、时序和验收基线上保持一致。

## 6. 非目标

以下内容不构成本文件所定义任务的目标：

- 仅做文档改名、索引调整或术语替换
- 仅完成 `modules/` 骨架搭建但继续把真实实现留在旧组织方式中
- 仅把旧架构降级为参考或兼容层而不做物理清除
- 借本轮任务进行与架构收敛无直接关系的开放式业务重定义
- 在本文件中给出实施步骤、技术选型结论或重构方案

## 7. 裁决口径

后续凡涉及任务边界、正式事实源或完成判定的争议，统一按以下口径裁决：

- `dsp-e2e-spec` 优先于任何历史说明、临时分析或残留材料。
- “全面迁移”按“所有模块同等级现实收敛”解释，不得退化为局部迁移。
- “舍弃旧仓架构”按“物理清除 legacy 资产”解释，不得降级为仅失去正式地位。
- 新架构是否成立，以仓库现实是否完成真实承载收敛为准，不以目录是否已经存在为准。
