# Design: modules 统一骨架对齐方案比较

更新时间：`2026-03-25`

## 1. 关键目标

- 让 `modules/` 下所有一级模块向统一骨架收敛。
- 让目录结构真实表达 `domain / services / application / adapters / runtime` 的职责边界。
- 让迁移方案兼容当前仓库的 `CMake`、测试和现有模块耦合现实。
- 先完成架构设计，再进入迁移实施。

## 2. 候选方案

### 方案 A：一次性全量硬切换

在单个迁移窗口内完成所有模块的目录重排、构建切换和旧结构退出。

### 方案 B：统一骨架先落位，保留兼容桥，分阶段收敛

先为所有模块建立统一骨架，再将现有实现和测试逐步迁入；旧结构在迁移期作为 bridge 保留，最终退出。

### 方案 C：按模块类型分治，对齐深度分级

根据模块复杂度和职责特性，定义不同迁移深度和节奏，优先推进风险较低的模块。

## 3. 对比

| 方案 | 复杂度 | 可维护性 | 性能/实时性扰动 | 主要风险 |
|---|---|---|---|---|
| A | 高 | 终态高，迁移期低 | 中 | 改动面过大，回归定位困难 |
| B | 中 | 终态高，迁移期中 | 低 | bridge 滞留、阶段性双结构 |
| C | 中 | 中到高 | 低 | 例外过多，统一性被削弱 |

## 4. 推荐结论

当前采用 `方案 B` 作为主方案。

## 5. 推荐理由

- 当前仓库的主要问题不是缺少模块 root，而是模块内部结构尚未统一，同时构建图已经与现结构绑定。
- 方案 B 能把骨架对齐、职责净化、测试迁移和构建切换拆成多个可验证阶段。
- 相比方案 A，方案 B 更符合当前仓库现实，能显著降低高复杂模块的迁移风险。
- 相比方案 C，方案 B 保留统一终态，不把“分级例外”固化成长期制度。
