# Feature Spec: modules 统一骨架对齐

更新时间：`2026-03-25`

## 1. 需求本质

本轮工作的本质是对 `modules/` 下全部一级模块执行结构治理，使模块目录不再停留在 `contracts + src + README + CMakeLists.txt` 的 owner 入口状态，而是进一步收敛到参考模板建议的统一骨架，用物理目录稳定表达职责边界、契约边界和后续治理入口。

## 2. 冻结目标

- 对齐对象是 `modules/` 下全部一级模块。
- 目标是向统一骨架收敛，而不是只处理单个模块。
- 当前阶段已确认采用统一骨架治理问题，而不是直接做一次性全量代码重写。
- 需求冻结后不得自行修改目标、假设与边界。

## 3. 前提假设

- 参考依据来自 `docs/architecture/dsp-e2e-spec/` 中关于模块统一骨架的建议。
- 当前工作区已经完成顶层 canonical roots 与模块 owner path 的收敛。
- 模块内部骨架尚未统一，现状与模板之间仍有明显差距。
- 后续设计与迁移必须建立在当前仓库真实结构和真实构建图之上。

## 4. 约束与边界

- 判断必须基于仓库事实、冻结文档和参考模板，不能基于主观偏好。
- 目录迁移必须考虑 `CMake`、测试入口、契约位置和历史耦合。
- 本轮设计的关注点是统一骨架对齐，不做与该目标无直接关系的开放式业务重定义。
- 模块统一骨架属于工程治理设计，正式冻结文档是否更新需单独推进。
