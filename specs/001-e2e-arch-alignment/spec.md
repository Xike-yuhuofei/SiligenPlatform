# Feature Specification: SS-dispense-align E2E Architecture Alignment

**Feature Branch**: `001-e2e-arch-alignment`  
**Created**: 2026-03-23  
**Status**: Draft  
**Input**: User description: "以 docs/architecture/点胶机端到端流程规格说明 为参考模板，对 SS-dispense-align 进行面向端到端流程的架构重构与规格对齐"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 建立统一端到端架构基线 (Priority: P1)

作为架构负责人，我需要把当前仓的点胶业务链路重构为一套与参考模板一致的端到端架构基线，使团队能够用同一套阶段定义、对象定义、模块边界和状态语义描述系统，而不是继续依赖历史实现习惯和口头约定。

**Why this priority**: 如果没有统一基线，后续的模块重构、接口收敛、测试补齐和验收放行都会继续发生边界漂移，主线任务无法稳定推进。

**Independent Test**: 只检查规格产物即可独立验证；当评审者能够从任务创建一路追到执行归档，并在每个阶段看到明确输入、输出、成功判定、失败判定和追溯键时，这个故事就已独立成立。

**Acceptance Scenarios**:

1. **Given** 当前仓存在历史演化形成的多段流程和多套命名，**When** 团队依据重构后的规格查看端到端流程，**Then** 每个业务阶段都能被映射到唯一目标阶段，并具有清晰的输入对象、输出对象和阶段职责。
2. **Given** 任一关键流程节点出现争议，**When** 评审者回查规格，**Then** 能明确判断该节点属于哪个阶段、由哪个模块负责、失败责任停在哪一层。

---

### User Story 2 - 收敛模块边界与对象责任 (Priority: P2)

作为模块负责人，我需要依据统一规格明确当前系统中哪些对象是事实产物、哪些模块是唯一 owner、哪些边界禁止越权，这样我才能在重构时知道该保留什么、拆分什么、迁移什么。

**Why this priority**: 没有对象 owner 和模块边界，架构重构会退化成目录调整或零散重命名，无法真正消除耦合与职责混写。

**Independent Test**: 只检查模块边界、对象责任和迁移决策表即可独立验证；当每个核心对象都能对应唯一 owner 模块，并且越权行为被显式禁止时，这个故事就具备独立价值。

**Acceptance Scenarios**:

1. **Given** 一个核心对象需要被多个模块消费，**When** 团队查看重构后的规格，**Then** 能明确识别唯一 owner 模块、允许的消费者以及禁止的越权改写行为。
2. **Given** 当前仓中存在历史模块职责重叠，**When** 团队执行规格对齐，**Then** 每项重构决策都会被标记为保留、调整、拆分、新增、合并或移除中的一种，并具有可追溯理由。

---

### User Story 3 - 建立可验收的状态与测试闭环 (Priority: P3)

作为测试与验收负责人，我需要让新的架构规格天然支持状态机、命令/事件、故障分层和验收矩阵回链，这样验收就能基于事实对象和状态结果，而不是依赖人工观察。

**Why this priority**: 架构重构若不能落到验收与回归层，系统即使结构更整齐，也无法证明端到端流程真的被规格化和稳定化。

**Independent Test**: 只检查状态机、验收场景和测试回链规则即可独立验证；当关键链路都能映射到阶段、模块、对象、状态和失败码时，这个故事即成立。

**Acceptance Scenarios**:

1. **Given** 一个任务离线规划通过但设备预检失败，**When** 验收方检查规格，**Then** 能区分离线通过与在线可执行通过，并找到阻断、恢复或终止的明确规则。
2. **Given** 一次执行出现首件拒绝、运行中故障或暂停恢复，**When** 验收方回链测试基线，**Then** 能明确对应状态迁移、允许命令、预期事件和放行条件。

---

### Edge Cases

- 当现有能力无法自然映射到参考模板中的单一阶段时，系统如何记录拆分、合并或保留旧边界的理由，避免重构过程中出现职责真空。
- 当同一对象在历史实现中被多个模块同时创建或改写时，系统如何确定唯一事实 owner，并避免迁移后继续出现越权写入。
- 当离线规划结果满足规则但在线设备状态不满足执行条件时，系统如何明确阻断发生在哪个阶段，而不是把两类问题混为一类。
- 当部分链路已经完成规格对齐而其他链路仍沿用旧语义时，系统如何保证阶段命名、状态语义和验收口径不出现双轨并存。
- 当历史测试只覆盖 happy path 时，系统如何补齐首件、预检阻断、故障恢复、人工决策和归档追溯场景，避免“结构重构完成但验收缺口仍在”。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须产出一套覆盖点胶端到端主链路的统一规格基线，至少覆盖任务创建、文件接入、几何标准化、拓扑/特征处理、工艺规划、坐标与对位补偿、路径/轨迹规划、执行前校验、执行、监控和归档追溯。
- **FR-002**: 系统必须为每个在范围内的业务阶段定义明确的输入对象、输出对象、成功判定、失败判定和最低追溯字段。
- **FR-003**: 系统必须为所有核心业务对象定义统一对象清单，并明确每个对象的业务含义、关键属性边界和上下游关系。
- **FR-004**: 系统必须为每个核心对象指定唯一事实 owner 模块，并明确允许的消费者、禁止的越权改写行为以及失败归责停留层级。
- **FR-005**: 系统必须为跨模块协作定义统一契约基线，使对象引用、版本语义、阶段上下文、失败归因和追溯主键在跨模块交互中保持一致。
- **FR-006**: 系统必须把运行控制骨架整理为清晰的状态模型，至少区分任务推进、执行控制、人工决策和恢复/回退等不同控制对象。
- **FR-007**: 系统必须规定所有关键状态迁移的触发意图与确认结果，并明确哪些迁移属于可重试、可恢复、需人工决策或不可恢复终止。
- **FR-008**: 系统必须明确区分离线规划通过与在线设备可执行通过，且任何一类通过都不得被视为另一类通过的替代。
- **FR-009**: 系统必须为关键执行场景定义统一验收口径，至少覆盖正常执行、预检阻断、首件放行/拒绝、运行中故障、暂停恢复、人工终止和归档完成。
- **FR-010**: 系统必须建立测试与验收回链规则，使每条关键测试都能回链到阶段编号、模块编号、对象类型、状态结果和失败码类别。
- **FR-011**: 系统必须给出面向单仓多模块的目标组织基线，明确应用入口、核心领域模块、共享层、样本资产和测试资产的归位原则。
- **FR-012**: 系统必须对当前仓与目标模板之间的差距建立显式差距清单，并为每一项差距给出保留、调整、拆分、合并、新增或移除中的一种处理决定。
- **FR-013**: 系统必须定义重构完成的判定基线，使团队能够基于文档一致性、边界闭合度、状态闭合度和验收可执行性判断是否达到下一阶段。
- **FR-014**: 系统必须保证重构后的规格仍能保留现有业务闭环所需的关键关联关系，包括任务、图纸、工艺、设备、执行记录和追溯信息之间的关联。
- **FR-015**: 系统必须明确本轮重构范围的边界，禁止把与端到端流程规格无直接关系的性能调优、硬件整定、界面风格修改或现场安全认证工作混入本特性范围。

### Scope Boundaries

- 本特性的范围包括：端到端阶段链重构、对象链重构、模块边界重构、状态与命令语义重构、验收基线重构，以及这些内容在仓级组织上的对齐。
- 本特性不包括：硬件控制参数整定、单次现场问题排障、设备安全认证、纯界面视觉改版、无业务价值的目录美化或与点胶主链无关的仓库清理。
- 本特性关注的是“系统应该如何按规格组织和验收”，不是对任意一个既有实现细节做局部修补。

### Assumptions & Dependencies

- **Assumption A1**: `docs/architecture/点胶机端到端流程规格说明` 目录中的文档集被视为本轮架构重构的参考模板来源。
- **Assumption A2**: 当前业务主链仍以“从图纸输入到设备完成点胶并形成追溯记录”为核心闭环，模板中的阶段划分可作为目标模型。
- **Assumption A3**: wave1-7 期间已积累的主线信息可被复用为现状证据，但不自动等于目标结构本身。
- **Dependency D1**: 需要能够访问当前仓中的现状文档、现有模块划分和历史验收依据，以完成差距识别和迁移决策。
- **Dependency D2**: 需要架构、业务、测试三类评审角色对统一阶段命名、对象命名和验收口径达成一致，否则无法宣告规格闭环。
- **Dependency D3**: 需要参考模板中的阶段、模块、状态与验收文档在本轮期间保持稳定基线，否则重构目标会持续漂移。

### Key Entities *(include if feature involves data)*

- **Stage Specification**: 描述一个端到端业务阶段的规范单元，包含阶段目标、输入对象、输出对象、成功判定、失败判定、追溯字段和边界说明。
- **Artifact Definition**: 描述端到端流程中的核心业务对象，如任务、图纸、规范几何、工艺计划、执行包、设备快照和执行记录，并规定其 owner、版本语义和上下游关系。
- **Module Contract**: 描述一个模块对外承担的职责、可生成对象、可消费对象、禁止越权行为、交互契约和失败归责范围。
- **State Model**: 描述任务推进、执行控制、人工决策和回退恢复的状态集合、迁移规则、触发命令和确认结果。
- **Acceptance Baseline**: 描述验收与回归所需的场景清单、通过标准、阻断标准、回链字段和可复核证据。
- **Gap Resolution Item**: 描述当前仓与目标模板之间的一条差距及其处理决定、影响范围、责任归属和完成判定。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% 的在范围端到端阶段都能被映射到唯一目标阶段，并且每个阶段都具备输入、输出、成功判定、失败判定和最低追溯字段。
- **SC-002**: 100% 的在范围核心业务对象都具有唯一事实 owner、明确消费者边界和禁止越权规则，不再存在“多个模块同时拥有同一事实对象”的未决项。
- **SC-003**: 100% 的关键执行场景都具备明确验收场景与判定口径，至少覆盖正常执行、预检阻断、首件放行/拒绝、运行中故障、暂停恢复、人工终止和归档完成。
- **SC-004**: 所有在范围跨模块交互都能回链到统一的对象引用、阶段上下文和失败归因规则，正式评审中不再出现阻断级的跨文档命名冲突或责任冲突。
- **SC-005**: 100% 的现状差距都被归类并给出处理决定，且每项决定都具备责任归属和完成判定，避免出现“已识别但无人接手”的架构悬空项。
- **SC-006**: 架构、业务和测试评审者能够在一次评审中独立完成从任务创建到执行归档的端到端回链，不依赖额外口头解释即可定位阶段、对象、模块和验收关系。
