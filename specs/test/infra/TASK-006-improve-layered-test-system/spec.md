## User Scenarios & Testing *(mandatory)*

### User Story 1 - 按风险分层执行验证 (Priority: P1)

作为开发人员或代码审查门禁维护者，我需要一套按风险和成本分层的测试体系，使常规改动优先经过快速、离线、可重复的验证，再按需要进入更重的回归、性能和真实硬件验证。

**Why this priority**: 这是测试体系的主价值。如果无法先把多数改动收敛到低成本分层验证，后续的覆盖率、稳定性和联机安全都无法成立。

**Independent Test**: 仅实现这一故事时，应能让一个常规改动被正确路由到合适的验证层，并产出明确的通过、失败或升级验证结论。

**Acceptance Scenarios**:

1. **Given** 某次改动只影响纯逻辑或契约输出，**When** 发起验证，**Then** 系统应优先执行快速且离线的验证层，并给出是否需要继续升级验证的明确结论。
2. **Given** 某次改动影响执行链、状态机或设备相关行为，**When** 发起验证，**Then** 系统应在通过离线门禁后再安排更高成本的回归、性能或真实硬件验证。

---

### User Story 2 - 复用统一测试资产与故障场景 (Priority: P2)

作为测试维护者，我需要统一的样本资产、夹具、故障注入和分类口径，以便新增测试时复用既有能力，而不是为每个模块重复发明执行方式和证据格式。

**Why this priority**: 没有统一资产和夹具，测试会迅速碎片化，覆盖面看似增加，实际不可维护、不可比较、不可追溯。

**Independent Test**: 仅实现这一故事时，应能为一个新模块新增测试并复用统一样本、统一夹具和统一证据输出，而不需要单独定义新的测试运行方式。

**Acceptance Scenarios**:

1. **Given** 某模块需要新增一个正常场景和一个故障场景测试，**When** 测试维护者配置测试资产，**Then** 这两个测试应能复用统一的样本、故障注入和证据输出规则。
2. **Given** 多个模块消费同一类基线样本或事件轨迹，**When** 执行测试，**Then** 结果应可跨模块比较，且失败证据格式保持一致。

---

### User Story 3 - 用模拟和受限联机获得可诊断证据 (Priority: P3)

作为故障调查人员或联机验证人员，我需要多数高风险场景先在模拟全链路环境中自动复现，只有模拟无法证明的风险才进入受限真实硬件验证，并且每次失败都能留下可诊断证据。

**Why this priority**: 真实硬件验证成本高、风险高，必须让其成为最后门禁而不是第一现场，同时保证失败后能回溯到具体场景、状态与异常点。

**Independent Test**: 仅实现这一故事时，应能自动运行至少一条模拟全链路场景和一条受限真实硬件场景，并分别生成结构化证据与中止条件记录。

**Acceptance Scenarios**:

1. **Given** 模拟环境可以表达设备未就绪、超时或中断，**When** 执行全链路场景，**Then** 系统应自动给出阻断、恢复或回滚结果，并附带事件与状态证据。
2. **Given** 某场景必须依赖真实硬件才能确认，**When** 进入受限联机验证，**Then** 系统应要求先满足离线门禁，并记录观察点、停机条件和执行结论。

---

### Edge Cases

- 当测试场景同时涉及多个验证层时，系统必须选择最低且足够证明风险的验证层作为首选入口，避免无必要地直接跳到高成本验证。
- 当样本资产缺失、损坏或与当前契约不匹配时，系统必须把失败归因为测试资产问题，而不是把它误判为被测能力回归。
- 当同一故障在模拟环境和真实硬件环境中结论不一致时，系统必须保留两侧证据并标记为待诊断分叉，而不是自动覆盖较早结论。
- 当某条测试出现不稳定结果时，系统必须显式暴露其不稳定状态，禁止将其视为可信门禁结果。
- 当真实硬件资源不可用时，系统必须给出哪些风险仍可通过离线验证覆盖、哪些风险被明确延后，而不是输出模糊完成结论。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须定义清晰的验证分层，使结构门禁、纯逻辑验证、契约验证、离线集成验证、模拟全链路验证、性能验证和受限真实硬件验证具备明确边界与升级顺序。
- **FR-002**: 系统必须支持按改动风险、验证成本、环境依赖和证据价值对测试进行分类，并允许据此选择合适的执行层。
- **FR-003**: 系统必须提供统一的测试入口，使开发人员、CI 门禁和专项验证人员能够以一致方式触发不同层级的测试。
- **FR-004**: 系统必须为每条测试声明其主要 owner、目标风险、必需环境、执行成本和预期证据类型。
- **FR-005**: 系统必须优先让可离线验证的改动在离线环境中完成验证，不得将真实硬件作为默认第一验证环境。
- **FR-006**: 系统必须支持统一管理测试样本、基线产物、事件轨迹和异常输入，并允许跨模块复用这些测试资产。
- **FR-007**: 系统必须提供可复用的测试夹具，用于表达外部依赖、设备反馈、时序控制和故障注入，而不要求每个模块自行重建同类能力。
- **FR-008**: 系统必须支持注入关键故障场景，包括未就绪、超时、中断、重复事件、乱序反馈、回滚和恢复，以验证系统在异常条件下的可控行为。
- **FR-009**: 系统必须允许模块 owner 为其语义输出建立契约验证，确保下游依赖的输入、输出、错误分类和观测信号保持稳定。
- **FR-010**: 系统必须支持跨模块链路验证，以确认上游产物、执行准备、运行态控制和状态反馈能够按既定语义正确衔接。
- **FR-011**: 系统必须为关键业务链提供模拟全链路场景，至少覆盖成功、阻断、回滚、恢复和归档等代表性结果。
- **FR-012**: 系统必须将真实硬件验证限制在模拟无法充分证明的高风险场景，并要求每次联机验证都附带观察点、执行边界和中止条件。
- **FR-013**: 系统必须支持性能验证，能够比较常规样本、边界样本和长时运行场景下的关键基线变化。
- **FR-014**: 系统必须输出统一的结构化证据，至少包含执行摘要、失败分类、关键事件、状态快照和关联样本信息。
- **FR-015**: 系统必须对失败给出可诊断结论，明确区分被测能力回归、测试资产失效、环境依赖缺失和验证前置条件未满足。
- **FR-016**: 系统必须支持分层运行策略，使快速门禁、完整门禁、夜间广覆盖和专项联机验证能够使用同一套测试资产和分类口径。
- **FR-017**: 系统必须显式标记测试稳定性状态，并防止不稳定测试在未被识别与治理前承担正式门禁职责。
- **FR-018**: 系统必须在任务收尾时汇总已执行的验证层级、跳过原因、残余风险和文档影响，保证结论可追溯。

### Key Entities *(include if feature involves data)*

- **Validation Layer**: 一类具有固定验证目标、环境要求、成本边界和升级顺序的测试层级。
- **Test Asset**: 被多个测试复用的样本输入、基线产物、事件轨迹或故障数据。
- **Test Fixture**: 用于模拟依赖、控制时序、注入故障和采集证据的复用型测试支撑能力。
- **Execution Lane**: 面向不同场景的运行通道，如快速门禁、完整门禁、夜间广覆盖和受限联机验证。
- **Evidence Bundle**: 一次测试执行后产出的统一证据集合，用于判断通过、失败、阻断和待诊断状态。
- **Fault Scenario**: 经过命名和归类的异常或边界情形，用于验证系统在非理想条件下的行为。

## Assumptions

- 当前仓库已有基础构建与测试入口，本特性是在现有入口之上建立统一分层口径，而不是重新定义所有基础执行方式。
- 各业务模块继续对其语义输出负责，测试体系负责验证与收敛这些契约，不负责重新定义业务 owner。
- 真实硬件资源有限且风险较高，因此默认策略应是“先离线、后联机”。
- GUI 或展示层验证只承担用户可见入口确认，不承担大多数正确性判定职责。

## Scope Boundaries

- 本特性关注测试体系的分层能力、资产治理、执行策略和证据输出，不直接改变业务功能、设备语义或运行时状态机规则。
- 本特性不追求用真实硬件覆盖所有场景，而是优先扩大离线和模拟环境的可验证范围。
- 本特性不把界面自动化作为主要正确性判定手段，核心正确性应由契约、状态和链路验证承担。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 至少 95% 的常规代码改动能够在不依赖真实硬件的前提下完成首轮验证并得到明确结论。
- **SC-002**: 面向常规改动的快速门禁在 15 分钟内给出结果，且能明确指出是否需要升级到更高成本验证层。
- **SC-003**: 100% 的失败测试执行都能产出统一结构化证据，足以区分回归、环境缺失、资产失效和前置条件未满足。
- **SC-004**: 所有关键业务链都至少具备一条模拟全链路自动化场景，覆盖成功、阻断、回滚、恢复和归档五类代表性结果。
- **SC-005**: 所有被认定为关键 owner 的模块都具备可执行的契约验证，且其主要语义输出拥有基线比较依据。
- **SC-006**: 真实硬件验证只承担高风险、模拟不可充分证明的场景，且每次执行都记录观察点和中止条件，覆盖率达到 100%。
- **SC-007**: 夜间或专项广覆盖运行能够覆盖边界样本、故障矩阵和长时稳定性场景，并在单次运行后生成可追溯总结。
- **SC-008**: 正式门禁中的未治理不稳定测试占比低于 2%，且所有不稳定测试都被显式标记并排除在可信门禁之外。
