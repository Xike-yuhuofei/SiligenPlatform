# 07 Guide Test Case Design

## 1. Purpose

### 1.1 文档目的
本文档定义 `SiligenPlatform` 项目中正式测试用例的设计方法，用于统一：
- 如何从模块职责推导测试用例
- 如何选择样本类型与故障注入点
- 如何为不同测试层级设计可判定用例
- 如何避免“测试很多但无法稳定证明问题”的情况

### 1.2 与基准文档的关系
本文档是执行指南，不替代以下基准文档：
- `01-baseline-architecture.md`
- `03-baseline-module-test-matrix.md`
- `04-baseline-fixture-golden-evidence.md`
- `05-baseline-suite-label-ci.md`
- `06-baseline-simulated-line-and-hil.md`

当本文档与基准文档冲突时，以基准文档为准。

## 2. General Design Principles

### 2.1 单一判定目标
一个正式测试用例应尽量只证明一个核心命题，例如：
- 某个输入契约是否成立
- 某个状态迁移是否非法
- 某个故障恢复路径是否正确
- 某个输出载荷是否与 golden 一致

禁止一个测试同时承担多个主判定目标，否则失败时将无法快速定位责任层。

### 2.2 可重复执行
正式测试必须尽可能满足以下条件：
- 可在同一环境重复执行得到相同结论
- 不依赖人工交互
- 不依赖未固定的时序窗口
- 不依赖随机数据作为主判定依据

若测试天然依赖时间或并发，应显式固定时钟、超时阈值、并发规模和执行 profile。

### 2.3 可诊断失败
设计测试时，必须先设计失败证据，而不是只设计“如何跑通”。
每个测试用例在失败时，至少应回答：
- 失败发生在哪一层
- 哪个输入、事件或状态触发失败
- 哪个判定物不一致
- 下一步应检查什么

### 2.4 可沉淀为资产
能沉淀为长期资产的测试优先级更高。优先建设：
- 可复用 fixture
- 可复用 golden
- 可复用故障注入模板
- 可复用 simulated-line 场景骨架

### 2.5 由模块职责倒推测试
测试设计必须从模块职责出发，而不是从“代码实现细节可否方便 mock”出发。
优先问：
1. 该模块对外承诺什么
2. 该模块最容易把什么搞错
3. 下游最依赖它的什么语义
4. 发生失败时应在何处被最早发现

## 3. Sample Design Strategy

### 3.1 `small`
`small` 样本用于证明最小功能闭环成立，特点：
- 输入简单
- 判定物短小明确
- 失败定位成本低
- 适合作为 smoke 或最小 golden

例：
- 一个矩形 DXF
- 一个最小 payload
- 一个最小状态迁移序列

### 3.2 `normal`
`normal` 样本用于模拟真实业务常规输入，特点：
- 接近生产主路径
- 不刻意追求边界
- 用于验证主干正确性

例：
- 常规 bra DXF
- 正常 runtime 启动配置
- 正常 success acceptance 场景

### 3.3 `dirty`
`dirty` 样本用于模拟质量较差但现实存在的输入，特点：
- 格式合法但质量差
- 可能含噪声、冗余、重复、轻度损坏
- 重点验证鲁棒性与降级行为

例：
- 含重复段、微小退化段的 DXF
- 多余字段或旧字段混杂的协议样本
- 带历史遗留配置键的配置文件

### 3.4 `boundary`
`boundary` 样本用于证明边界规则稳定，特点：
- 极小/极大/极限参数
- 约束边缘
- 容差边缘
- 状态机临界路径

例：
- 极短段导致规划困难的 path
- 极限 jerk/acc 组合
- stop 与 pause 竞争的事件序列

### 3.5 `failure`
`failure` 样本用于验证系统显式失败而不是“带病成功”，特点：
- 输入或环境明确不满足约束
- 预期结论是失败、阻断、拒绝或回滚
- 需配套失败分类与证据要求

例：
- 缺失关键配置
- not-ready 设备状态
- 超时、断连、重复响应

## 4. Contract Test Design

### 4.1 输入契约设计
输入契约测试应覆盖：
- 必填字段缺失
- 非法值
- 边界值
- 合法但可疑值
- 旧版本/新版本兼容值

输入契约测试不应只测“happy path”，必须显式覆盖拒绝路径。

### 4.2 输出契约设计
输出契约测试应冻结：
- 字段集合
- 语义含义
- 顺序规则
- 默认值规则
- 版本行为

若输出结构适合 golden，应优先使用结构化 golden，而不是散落断言。

### 4.3 错误契约设计
错误契约测试应关注：
- 失败是否发生在正确层级
- 失败分类是否稳定
- 错误消息是否具备诊断信息
- 是否出现“错误被吞掉后假成功”

### 4.4 观测契约设计
观测契约测试用于证明：
- 日志键名稳定
- 事件名称稳定
- status/export 字段稳定
- timeline 能完整反映关键事实

对 runtime、workflow、HMI 等模块，观测契约通常与功能契约同样重要。

## 5. State Machine Test Design

### 5.1 状态定义覆盖
必须先列出状态全集，再设计测试。状态不明确时，禁止直接写状态机测试。
测试至少覆盖：
- 初始态
- 目标终态
- 中间关键态
- 失败态
- 恢复态

### 5.2 事件覆盖
为每个状态列出允许事件、禁止事件和无效事件。
测试至少应覆盖：
- 合法触发
- 非法触发
- 重复触发
- 竞争触发

### 5.3 非法迁移覆盖
非法迁移测试必须显式存在，不能用“未写即默认不会发生”代替。
重点覆盖：
- skip 中间态直达终态
- 失败态直接假恢复
- stop/pause/resume/reset 的优先级错误

### 5.4 恢复路径覆盖
只验证失败是不够的，还必须验证：
- 失败后是否可恢复
- reset 后是否回到正确起点
- 恢复后是否丢上下文
- 恢复失败时是否再次正确失败

### 5.5 幂等与重入覆盖
对 runtime、workflow、service 宿主尤为关键。
测试应覆盖：
- 重复提交
- 重复 stop
- 重复 ack
- inflight joined / superseded
- 断连后重入

## 6. Failure Injection Test Design

### 6.1 同步故障
同步故障指在单次调用或单步处理过程中立即出现的故障，例如：
- invalid-input
- missing-config
- mode-mismatch

这类测试应优先放在模块内，并给出明确错误契约。

### 6.2 异步故障
异步故障指在系统运行过程中由外部事件或时间推进触发的故障，例如：
- disconnect-mid-flight
- feedback-missing
- duplicate-response
- out-of-order-response

这类测试通常需要事件录制和时间控制。

### 6.3 时间类故障
时间类故障必须固定：
- timeout 阈值
- retry 次数
- retry 间隔
- 上下文保留规则

禁止让“机器偶尔慢一点”成为唯一的超时触发手段。

### 6.4 顺序类故障
顺序类故障重点验证：
- 乱序消息
- 早到/晚到事件
- 多源事件冲突
- 状态已过期但旧事件继续到达

### 6.5 模式类故障
模式类故障重点覆盖：
- real/mock 模式切换不一致
- online/offline 状态投影错误
- HIL/simulated-line profile 不一致

## 7. Integration Test Design

### 7.1 链路最小闭环
integration 测试的基本策略是：
- 尽量使用最小链路闭环
- 只引入为验证边界所必需的上下游
- 不把 integration 退化为伪 E2E

### 7.2 上下游边界点
integration 测试应重点放在这些边界点：
- job-ingest -> dxf-geometry
- dxf-geometry -> 上游特征/工艺层
- process-path -> motion-planning
- motion-planning -> dispense-packaging
- runtime-service <-> runtime-execution

### 7.3 共享事实样本
integration 应优先复用共享 `samples/`、`fixtures/`、`golden/`，避免每条链各自维护样本副本。

### 7.4 不可见状态的观测方法
若集成链路中某些内部状态不可直接断言，应通过以下手段观测：
- 结构化 event
- status export
- artifact dump
- timeline 关键锚点

## 8. Acceptance Scenario Design

### 8.1 `success`
用于证明主干闭环成立。至少应证明：
- 输入可被接受
- 主链可完成
- 最终态正确
- 证据完整

### 8.2 `block`
用于证明不满足前提时系统明确阻断而非带病执行。重点覆盖：
- not-ready
- precondition missing
- invalid mode

### 8.3 `rollback`
用于证明中途失败时系统能退回安全状态，至少覆盖：
- 部分动作已发生
- 需要显式撤销或终止
- 用户/宿主可感知结果

### 8.4 `recovery`
用于证明故障后可恢复。至少覆盖：
- reset
- reconnect
- resume
- re-enter ready state

### 8.5 `archive`
用于证明任务结束后证据、状态、artifact 生命周期正确收束，而不是只停留在“执行结束”。

---

## 9. Performance Test Design

### 9.1 `cold`
用于证明首次运行路径的延迟、资源开销与初始化行为可接受。

### 9.2 `hot`
用于证明 warm cache / warm service 条件下的重复运行行为稳定。

### 9.3 `singleflight`
用于证明 authority/inflight dedup 行为正确，重点不是仅看性能，还要看语义正确性。

### 9.4 `stress`
用于证明在高负载或资源逼近上限时，系统仍能给出可诊断结果，而不是静默损坏。

### 9.5 `soak`
用于证明长时间运行下：
- 没有明显泄漏
- 没有状态腐化
- 没有报告/证据质量衰减

## 10. Test Review Checklist

### 10.1 设计评审清单
- 该测试要证明什么是否单一且清晰
- 测试层级是否正确
- 是否应该下沉到更低层
- 样本是否覆盖了真正风险点

### 10.2 判定物评审清单
- 是否有稳定断言
- 是否有结构化 golden
- 是否有动态字段掩码规则
- 失败后能否看出哪里错

### 10.3 证据评审清单
- 是否输出最小 evidence bundle
- 是否能支持复盘
- 是否能支持机器对比
- 是否便于 CI 归档

## 13. Normative Rules

### 13.1 MUST
- 每个正式测试必须有清晰主判定目标。
- 每个 failure 类测试必须有明确的失败预期和证据要求。
- 状态机测试必须先有状态与事件清单。
- acceptance 场景必须对齐 `success / block / rollback / recovery / archive` 语义。

### 13.2 SHOULD
- 优先用结构化 golden 与 evidence，而不是散落断言。
- 优先用 lower layer 测试证明问题，再考虑更高层。
- 尽量沉淀共享样本与共享故障模板。

### 13.3 MUST NOT
- 不得将一个高层 E2E 用例当作多个低层契约缺失的替代品。
- 不得设计只能靠人工解释日志才能判定结果的正式测试。
- 不得以“真实机器偶发失败”代替正式故障注入用例。
