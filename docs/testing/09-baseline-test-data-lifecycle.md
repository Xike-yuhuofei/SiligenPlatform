# 09 Baseline Test Data Lifecycle

## 1. Purpose

### 1.1 文档目的
本文档冻结测试数据资产的生命周期治理规则，统一：
- fixture / golden / performance baseline / HIL capture 等测试资产的分类
- 新增、评审、更新、弃用、删除规则
- 版本化与漂移治理
- 大样本与敏感数据管理策略

### 1.2 为什么测试数据需要治理
本项目测试体系依赖大量可复用样本、golden 与结构化证据。

若无生命周期治理，容易出现：
- 样本来源不明
- golden 被无说明覆盖
- baseline 漂移不可追溯
- 大样本管理失控
- 历史测试无法复现

---

## 2. Asset Categories

### 2.1 fixture
`fixture` 是测试输入资产，包括但不限于：
- DXF 样本
- 配置样本
- 协议样本
- 故障样本
- 标定参数样本
- 宿主启动样本

### 2.2 golden
`golden` 是期望输出资产，包括但不限于：
- 结构化 JSON
- 文本 dump
- payload dump
- 路径规划摘要
- 状态迁移摘要

### 2.3 performance baseline
`performance baseline` 是用于性能回归比较的基准数据，包括：
- 指标摘要
- baseline profile 元数据
- 样本版本信息
- 环境元数据

### 2.4 HIL capture
`HIL capture` 是来自硬件闭环或硬件 smoke 的证据资产，包括：
- 时序抓取
- 事件流摘要
- 诊断日志
- 协议抓包摘要
- 传感/反馈记录

### 2.5 reference reports
`reference reports` 是用于说明某一 profile、场景或里程碑的参考报告，不直接作为 golden，但可作为审查依据。

---

## 3. Asset Lifecycle

### 3.1 create
创建新资产时必须声明：
- 资产类型
- 来源
- 归属模块或验证面
- 使用场景
- 是否进入正式基准

### 3.2 review
进入正式测试体系前，资产必须经过 review，至少确认：
- 来源可信
- 语义清晰
- 与现有资产不冲突
- 命名、位置与元数据满足规范

### 3.3 accept
被 accept 的资产方可进入：
- 正式 fixture 集
- 正式 golden 集
- 正式 performance baseline 集
- 正式 HIL capture 归档集

### 3.4 version
被 accept 的资产必须具有可追踪版本语义。版本可以通过：
- 文件名版本
- 元数据版本
- 目录版本
- schema 版本
进行表达。

### 3.5 deprecate
当资产不再适合作为正式基准时，应先标记为 `deprecated`，而不是直接删除。

### 3.6 remove
只有在满足删除条件时，方可从正式资产集中移除。删除必须保留替代关系或删除原因说明。

---

## 4. Versioning Policy

### 4.1 文件版本
对稳定长期资产，建议采用显式版本号或日期化版本标识。

### 4.2 schema 版本
若资产依赖结构化 schema，则 schema 版本必须可追踪，避免出现：
- 同名不同义
- 不同结构复用同一 golden
- parser 与 golden 不一致

### 4.3 协议样本版本
协议样本必须显式标识：
- 协议版本
- 方向（入站/出站）
- 是否为兼容样本
- 是否为故障样本

### 4.4 大样本版本
大样本必须至少具备：
- 来源说明
- 规模说明
- 采样或裁剪说明
- 版本标识

---

## 5. Update Governance

### 5.1 谁可以更新
正式测试资产只能由满足以下条件的变更更新：
- 变更说明清晰
- 责任 owner 明确
- 有对应代码或基准文档支撑
- 有审查记录

### 5.2 什么情况允许更新
允许更新资产的情况包括：
- 业务语义正式变化
- 协议版本正式变化
- 样本版本升级
- 历史资产被证明错误或失真
- 环境依赖已发生正式变更

### 5.3 更新必须附带什么证据
更新正式资产时必须附带：
- 变更原因
- 影响范围
- 对应实现或文档依据
- 新旧差异说明
- 必要的对比证据

### 5.4 baseline 漂移审核
对 golden 或 performance baseline 的更新，必须进行漂移审核，明确回答：
- 这是预期变化还是异常变化
- 为什么旧 baseline 不再适用
- 新 baseline 是否已稳定

---

## 6. Large Asset Policy

### 6.1 大样本定义
满足以下任一条件可视为大样本：
- 文件体积显著大于常规样本
- 解析或规划成本显著高于常规样本
- 只适合 nightly 或专项 profile 执行
- 需要单独缓存或下载策略

### 6.2 存储策略
大样本不应与常规样本无差别混放。应至少区分：
- 常规基准样本
- 夜间或专项样本
- 历史归档样本

### 6.3 裁剪策略
如使用裁剪或变体样本，必须记录：
- 原始来源
- 裁剪规则
- 保留哪些语义
- 为什么裁剪后仍可代表目标场景

### 6.4 缓存与下载策略
大样本如需下载或缓存，应确保：
- 下载入口稳定
- 版本固定
- 缓存可失效与可重建
- 缺失时的失败信息明确

---

## 7. Sensitive Data And Compliance

### 7.1 脱敏要求
若测试数据来自外部或业务环境，必须先评估是否需要：
- 去标识化
- 路径脱敏
- 客户/订单/设备信息脱敏
- 诊断日志关键字段脱敏

### 7.2 外部来源数据限制
外部来源数据不得未经审核直接进入正式 fixture 或 golden 集。

### 7.3 可公开与不可公开资产
应明确哪些资产：
- 可随仓库提交
- 仅限内部环境使用
- 只能通过受控方式访问

### 7.4 清理要求
对不再允许保留的测试数据，必须有明确清理路径与替代说明。

---

## 8. Deprecation Policy

### 8.1 弃用条件
资产可被标记弃用的条件包括：
- 上游 schema 已升级
- 业务语义已迁移
- 样本不再代表正式目标场景
- 历史资产存在已知缺陷

### 8.2 替代关系
弃用资产应尽量声明：
- 被哪一资产替代
- 替代后的差异
- 为什么替代更合适

### 8.3 历史兼容保留策略
对协议兼容、历史回放、迁移对比等场景，需要保留少量历史资产时，必须明确：
- 保留目的
- 保留时长
- 使用范围

---

## 9. Drift Detection

### 9.1 fixture drift
`fixture drift` 指输入资产发生变化但未正确反映其语义变化，导致测试结果不可比较。

### 9.2 golden drift
`golden drift` 指 golden 因覆盖式更新、动态字段污染或版本混乱而失去判定意义。

### 9.3 performance drift
`performance drift` 指 baseline 因样本、环境或 profile 语义变化而不能代表原先基准。

### 9.4 环境漂移
环境漂移包括：
- 运行时模式变化
- 依赖版本变化
- 硬件 profile 变化
- 路径和系统配置变化

测试资产治理必须考虑环境漂移，而不是只盯住文件本身。

---

## 10. Anti-Patterns

### 10.1 无版本样本
没有版本或来源信息的正式资产不可作为长期基准。

### 10.2 覆盖式更新 golden
直接覆盖 golden 而不记录差异，会使漂移不可审计。

### 10.3 来源不明的大样本
来源不清、不可复用或无法重建的大样本不得进入正式基准。

### 10.4 测试数据与文档脱节
若文档宣称存在某类样本，但仓库中无法定位或无法识别版本，则视为治理失败。

---

## 11. Normative Rules

### 11.1 MUST
- MUST 为正式测试资产定义类型、来源、版本与归属。
- MUST 对正式 golden 与 performance baseline 的更新进行漂移审核。
- MUST 对大样本提供来源、规模与使用说明。
- MUST 对需脱敏的测试数据执行脱敏后方可进入正式资产集。

### 11.2 SHOULD
- SHOULD 保持测试资产最小而有代表性。
- SHOULD 将历史兼容资产与当前主基准资产分开管理。
- SHOULD 对大样本使用清晰的缓存/获取策略。

### 11.3 MUST NOT
- MUST NOT 将来源不明或不可复现的资产纳入正式基准。
- MUST NOT 无说明覆盖正式 golden 或 baseline。
- MUST NOT 让测试资产长期处于“临时但被事实依赖”的状态。
