# 迁移原则

## 1. 目标

本文件定义从当前并列子项目形态过渡到根级 monorepo 形态时必须遵守的迁移原则。

## 2. 基本原则

1. 先立 canonical 路径，再做物理迁移。
2. 先冻结契约，再拆实现。
3. 先明确应用入口，再拆业务包。
4. 先保留兼容壳，再逐步淘汰 legacy 路径。
5. 不把 `build/` 产物当成源码事实。
6. 不因为“已有子项目”就放弃根级 canonical 落位。

## 3. 本阶段允许做的事

- 创建根级目录骨架
- 创建占位 README 和治理文档
- 新增统一工作区说明
- 为后续迁移固定 app/package/integration/tools/docs/config/data/examples 的稳定落位

## 4. 本阶段不允许做的事

- 删除现有子项目
- 一次性整体搬迁 `control-core`、`hmi-client`、`dxf-pipeline`
- 一步到位改写所有构建文件
- 在未冻结契约前大面积拆分业务实现

## 5. 迁移顺序

推荐执行顺序如下：

1. 工作区治理骨架
2. 契约层 owner 明确
3. 应用入口补位和对齐
4. 核心内核收拢
5. 宿主/设备/传输剥离
6. 集成验证统一收口

## 6. 兼容壳原则

兼容壳允许存在，但必须满足：

- 有明确替代的 canonical 路径
- 有退出条件
- 不再作为默认阅读入口
- 不继续承接新设计

## 7. 新增工作的落位原则

后续任务中，新增内容按以下规则判断：

- 新 app 壳层：落到 `apps/`
- 新共享能力：落到 `packages/`
- 新跨包场景：落到 `integration/`
- 新脚本或 codegen：落到 `tools/`
- 新治理与架构说明：落到 `docs/`
- 新稳定配置：落到 `config/`
- 新版本化资产：落到 `data/`
- 新示例：落到 `examples/`

## 8. 冲突处理原则

若根级 canonical 路径与现有子项目目录发生语义冲突：

- 优先新增或扩展 canonical 路径
- 保留旧目录作为兼容实现承载体
- 不通过删除旧目录来“制造整洁”

## 9. 完成标准

本阶段完成的标志不是“迁完了”，而是：

- 根级目录骨架已经建立
- WORKSPACE 规则明确
- 治理文档可指导后续任务落位
- 现有子项目与新 canonical 路径的兼容关系已经说清
