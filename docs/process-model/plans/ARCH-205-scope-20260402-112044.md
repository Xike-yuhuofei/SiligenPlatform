## 1. 任务归类结论

- 类型：`boundary-audit`
- 归类依据：
  1. 当前任务核心不是新增功能或单点故障修复，而是 `modules/dispense-packaging/` 的 owner、边界、依赖方向和目录落点治理。
  2. 审查文档已经给出明确结构偏移线索：模块文档、契约文档、构建入口与实际 live 调用链不一致。
  3. 当前最小可执行动作应先收敛“当前事实”和“阶段边界”，而不是直接跨模块迁移 owner。
- 风险标签：
  - `owner-drift`
  - `boundary-mismatch`
  - `mixed-live-and-parked`

## 2. 已知事实

1. `modules/dispense-packaging/README.md` 当前宣称 `domain/dispensing/` “只保留 packaging/validation 相关 domain 实现，不再编译 planner/optimizer 实现”。
2. `modules/dispense-packaging/domain/dispensing/CMakeLists.txt` 仍把 `PathOptimizationStrategy.cpp`、`TriggerPlanner.cpp`、`DispensingController.cpp`、`DispensingProcessService.cpp`、`AuthorityTriggerLayoutPlanner.cpp`、`CurveFlatteningService.cpp`、`TopologySpanSplitter.cpp`、`TopologyComponentClassifier.cpp` 等文件编进 `siligen_dispense_packaging_domain_dispensing`。
3. `modules/dispense-packaging/contracts/README.md` 仍把 `modules/workflow/application/usecases/dispensing/` 写成 canonical 契约实现面之一。
4. `modules/workflow/application/usecases/dispensing/PlanningUseCase.cpp` 与 `apps/runtime-service/container/ApplicationContainer.Dispensing.cpp` 继续直接依赖 `DispensePlanningFacade`。

## 3. 任务目标

1. 把 `dispense-packaging` 模块文档、契约文档和构建注释修正为当前真实 live 边界，不再宣称未完成的 owner 迁移已经完成。
2. 新增一份 live-vs-parked 清单，明确哪些源文件当前参与编译，哪些只存在于目录中但未进入 target。
3. 为后续阶段迁移建立受控起点，避免在 owner 尚未冻结时继续扩大 `DispensePlanningFacade` 或 `domain/dispensing` 的语义面。

## 4. 非目标

1. 不在本阶段迁移 `workflow`、`runtime-execution`、`apps/runtime-service` 的调用链。
2. 不修改 `ExecutionPackage*`、`TriggerPlan`、`AuthorityTriggerLayout` 等公开类型定义。
3. 不在本阶段删除 `domain/dispensing` 中的 residual 源文件。
4. 不把 `DispensePlanningFacade` 拆成多服务，不引入新的 public API。
