# ARCH-205 Architecture Plan - dispense-packaging Owner Boundary Phase 1

- 状态：Prepared
- 日期：2026-04-02
- 分支：`refactor/process/ARCH-205-dispense-packaging-owner-repair`
- 工作流上下文：`ticket=ARCH-205`，`timestamp=20260402-112044`
- 关联 scope：`docs/process-model/plans/ARCH-205-scope-20260402-112044.md`

## 1. 重构目标

1. 先恢复 `modules/dispense-packaging/` 作为 current-fact 模块入口的可信度，不再把未完成迁移写成已完成事实。
2. 在不改 consumer 调用链的前提下，显式标记 M8 当前的 live owner 面、residual live 源和 parked 源。
3. 为后续 owner 迁移建立阶段边界，避免在错误口径上继续叠加实现。

## 2. 分阶段计划

### 阶段 1
- 目标：修正模块内 current-fact 文档与构建注释，并新增 live-vs-parked 清单。
- 变更范围：
  - `modules/dispense-packaging/*.md`
  - `modules/dispense-packaging/module.yaml`
  - `modules/dispense-packaging/domain/dispensing/CMakeLists.txt`
  - `docs/process-model/plans/ARCH-205-*.md`
- 验证要求：
  - 文档口径与 `domain/dispensing/CMakeLists.txt`、`application/CMakeLists.txt` 的 source/include/link 事实一致。
  - 模块内不再出现“planner/optimizer 已不再编译”“workflow 是 canonical 契约 owner 面”等失真表述。
- 回退方式：
  - 仅回退本阶段文档与注释文件。
  - 不涉及源文件布局或 public 类型，回退不会影响运行语义。

### 阶段 2
- 目标：收缩 `DispensePlanningFacade` 的混合职责，并把 export/preview/execution assembly 责任拆出独立内部服务。
- 变更范围：
  - `modules/dispense-packaging/application/**`
  - 受影响的模块内测试
- 验证要求：
  - `dispense-packaging` 单测至少覆盖 façade 拆薄后的等价行为。
  - `workflow` 对 façade 的 consumer 行为不变。
- 回退方式：
  - 保持 façade public API 不变，仅回退内部服务拆分。

### 阶段 3
- 目标：迁出 `domain/dispensing` 中的 planning/process-control residual，实现 owner 真正收口。
- 变更范围：
  - `modules/dispense-packaging/domain/dispensing/**`
  - 可能波及 `workflow` / `runtime-execution` / 上游 planning owner
- 验证要求：
  - 必须先冻结目标 owner 模块与 authority artifact。
  - 必须有跨模块编译和直接回归验证。
- 回退方式：
  - 保留阶段性 bridge，但必须有显式退场计划。

## 3. 过渡期兼容策略

1. 阶段 1 明确承认 `workflow` 仍是 M8 application consumer，但不再把它写成契约 owner 面。
2. 阶段 1 不删除 residual 源文件，只通过 inventory 和文档把 live / parked / owner-questioned 状态显式化。
3. 在阶段 2 和阶段 3 开始前，不扩张 `DispensePlanningFacade` 的 public surface，不新增新的跨模块直连。

## 4. 剩余技术债

1. `DispensePlanningFacade` 仍是 preview、execution assembly、export request 组装的混合入口。
2. `domain/dispensing` 仍同时承载 packaging owner 与 planning/process-control residual。
3. `module.yaml` 的 `owner_artifact: ExecutionPackage` 与 README 中对 `DispenseTimingPlan` / `TriggerPlan` 的表述仍存在后续冻结需求。

## 5. 风险说明

1. 本阶段只修正文档和注释，不能被误解为 owner 迁移已经完成。
2. 如果后续阶段直接移动 residual 源文件而不先冻结 consumer 和 authority，会把架构治理变成行为回归风险。
3. `PUBLIC` include/link 暴露问题已被记录，但本阶段不触碰 ABI/consumer 编译面。
