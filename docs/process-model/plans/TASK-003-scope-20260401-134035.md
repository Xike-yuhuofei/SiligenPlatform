# TASK-003 Scope - estop_known Authority Under Disconnected State

- 状态：Prepared
- 日期：2026-04-01
- 分支：`fix/protocol/TASK-003-estop-known-authority`
- 工作流上下文：`ticket=TASK-003`，`timestamp=20260401-134035`
- 来源：从 `BUG-312` 被动断线收口中拆出的独立后续任务，不纳入 `BUG-312` 主修复提交

## 1. 任务归类结论

- 类型：`incident`
- 归类依据：
  1. 当前系统已经存在可观察异常：在 `connected=false`、`connection_state=disconnected`、`machine_state_reason=hardware_disconnected` 的状态下，`door_open_known=false`，但 `estop_known=true` 且 `effective_interlocks.sources.estop=system_interlock`。
  2. 这不是“新增能力”，而是现有 `status` 契约在断线态下的已知性口径不一致，可能误导 HMI、诊断和运维判断。
  3. 是否应继续对外宣称 `estop_known=true` 目前缺少显式契约冻结，因此必须单独受理，而不能并入已完成的被动断线修复。
- 风险标签：
  - `protocol-authority`
  - `safety-semantics`
  - `disconnected-state`

## 2. 已知事实

1. `tests/reports/online-validation/passive-disconnect-postfix-20260401-131941/l5-summary.json` 的未连接基线和被动断线快照都显示：
   - `connected=false`
   - `connection_state=disconnected`
   - `machine_state_reason=hardware_disconnected`
   - `door_open_known=false`
   - `estop_known=true`
   - `effective_interlocks.sources.estop=system_interlock`
2. `apps/runtime-gateway/transport-gateway/src/tcp/TcpCommandDispatcher.cpp` 当前存在如下已知组装逻辑：
   - `estop_known = connected || estop_state_known`
   - `effective_interlocks.sources.estop = system_interlock` 当 `estop_state_known=true`
3. `apps/hmi-app/src/hmi_client/client/protocol.py` 会把 `effective_interlocks.estop_known` / `io.estop_known` 作为 gate 口径的一部分继续向 UI 传播。
4. `shared/contracts/application/models/states.json` 当前要求 `io.estop_known` 与 `effective_interlocks.estop_known` 均为必填字段，但未在现有仓内文档中明确“断线态下 estop known 的 authority artifact 是什么”。

## 3. 任务目标

1. 冻结并澄清 `estop_known` 在 `Disconnected` / `hardware_disconnected` 场景下的 authority 口径。
2. 消除 `door_known=false` 但 `estop_known=true` 这类缺少契约说明的断线态不对称已知性。
3. 确保 `runtime-gateway status`、共享契约和 HMI consumer 对 `estop_known` 的解释一致，不产生双重真值。

## 4. 非目标

1. 不修改 `disconnect` RPC 的幂等确认语义。
2. 不重新打开 `BUG-312` 已闭环的“被动断线后长期 degraded”主故障。
3. 不把 `door_open_known`、home boundary、所有 interlock source 一次性打包重构。
4. 不在本任务中引入真实轴运动、真实阀输出或扩大联机验证范围。

4. 全量安全架构重构

## 7. 成功标准

1. 明确写出 `estop_known` 的 semantic object、authority artifact、owner layer 和 consumer-only layers。
2. 明确 `Disconnected`、`Connected`、`Degraded`、采样不可用等状态下 `estop_known` 应如何收敛。
3. `status` 契约、gateway 组装逻辑、fixtures、协议解析与直接消费测试保持一致。
4. 断线态不再输出“看似已知、但 authority 未冻结”的 `estop_known` 结果。
5. 若结论是“断线态下仍可合法 known”，则必须有显式文档和测试证明该 authority 仍有效，而不是隐式沿用当前实现。

## 8. 预期验证层级

1. `L0`：契约/owner/字段语义一致性检查
2. `L1`：gateway status 组装单测、共享契约测试、HMI 协议解析单测
3. `L2`：必要时补最小集成验证，确认 disconnected snapshot 与 consumer 解释一致
4. `L5`：默认不做；只有离线无法判定且需要真实来源佐证时才受限联机

## 9. 停止条件

1. 如果推进后发现核心问题其实是 `disconnect` RPC 语义，而不是 `estop_known` authority，则停止并拆成独立任务。
2. 如果推进后发现 owner 不在 `protocol/runtime-gateway`，而在更上游 `system_interlock` 领域定义层，则先做 `contract-freeze`，不要直接改 gateway。
3. 如果任务扩展到“所有 known 字段统一治理”，则停止当前路径并重新划分范围。

## 10. 证据缺口

1. 当前仓内没有显式文档定义“断线态下 estop 是否仍可 known”的权威口径。
2. 当前没有直接针对 `estop_known` disconnected 语义的最小单测或契约测试。
3. 目前只确认了现象和传播路径，尚未冻结 authority artifact。
