# 无机台强执行模板

```text
请在当前仓库中，作为测试执行者而不是顾问，直接对指定功能做“无机台验证”，并推进到拿出明确结论为止。

测试范围：
1. 点动
2. 回零
3. 点胶阀控制
4. 供胶控制
5. 急停
6. move/move_to
7. 安全门、急停、限位对上述功能的互锁拦截

硬约束：
1. 不使用真实机台，不连接真实硬件。
2. 必须优先复用仓库内已有的 mock server、smoke 脚本、单元测试、集成测试、ctest、协议测试入口。
3. 不要停留在“告诉我怎么做”，你要直接执行。
4. 不要因为第一次命令失败就停止；要继续定位、修复路径、补构建产物、切换正确入口后再执行。
5. 不要默认全量重编；如需构建，只构建最小必要目标。
6. 先不要改业务代码；只有当现有测试覆盖明显不足时，才允许补最小测试代码或测试脚本。
7. 全程使用简体中文。

执行策略：
1. 先扫描仓库，定位每个功能相关的：
   - HMI/UI 入口
   - 协议层入口
   - mock server
   - smoke 脚本
   - 单元测试/聚合测试/ctest 入口
   - 领域层核心实现
2. 每项功能都尽量按以下顺序验证：
   - 最接近真实链路的 HMI 或在线 smoke
   - 协议层 + mock 联调
   - 领域层/规则层单元测试
3. 每项功能至少覆盖：
   - 正常路径
   - 前置条件检查
   - 互锁拦截
   - stop/release/close 类收尾动作
4. 如果测试被聚合在大型测试二进制中，不允许因为“ctest 名称不直接对应”就跳过；必须定位真实二进制、构建目标和过滤方式并实际执行。
5. 如果因为缺少测试产物无法运行：
   - 先查 CMake / CTest / build 目录
   - 再补构建最小目标
   - 再执行目标测试
6. 如果仓库里完全没有某项功能的现成可执行测试入口：
   - 明确指出缺口
   - 补最小必要测试
   - 实际执行补充测试
7. 只有在确实无法推进时，才允许把该项列为阻塞，并说明已经尝试过什么。

过程要求：
1. 先发简短进度说明，再开始执行。
2. 执行过程中持续汇报你正在测什么、用了什么入口、遇到了什么问题、如何处理。
3. 不要泛泛而谈，所有结论必须基于实际命令输出或代码定位结果。
4. 如果发现“测试存在但当前工作区没构建”，要继续把它构建出来并跑掉，而不是只提示我。

最终交付要求：
最后输出正式测试报告，必须包含以下四部分：

一、已通过
对每个功能写清楚：
1. 使用了哪些命令/脚本/测试入口
2. 实际结果
3. 结论
4. 证据位置

二、失败或异常
对每个失败项写清楚：
1. 卡在哪一步
2. 实际报错
3. 已尝试的处理
4. 当前判断是代码问题、测试问题还是环境问题

三、未覆盖项
明确列出当前无机台测试仍然无法证明的内容，例如：
1. 电机真实运动方向
2. 真实 IO 接线
3. 伺服响应
4. 真实限位/急停硬件链路
5. 停止距离、抖动、过冲、丢步等真实运动行为

四、汇总表
表格列必须包含：
1. 功能
2. 测试层级（HMI/协议/领域）
3. 是否执行
4. 是否通过
5. 证据
6. 风险/未覆盖项

补充要求：
1. 如果某项功能其实和点动复用了同一条链路，要明确说明“共用了哪一层，额外验证了哪部分差异”。
2. 如果 mock 行为和真实后端可能存在偏差，要明确标出来，不要把 mock 通过等同于真实上线通过。
3. 最终结论必须明确，不要使用“可能”“大概”“应该没问题”这类表述。
```
