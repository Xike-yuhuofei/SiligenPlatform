# 首层业务场景逐项说明稿

## 文档定位

本稿基于已冻结的《工程替身系统总纲说明页》《需求定义稿》《非目标边界》《成立条件稿》《范围定义稿》《核心范围分层稿》和《首层场景筛选稿》，只回答一个问题：

**`S1-S9` 各自到底在证明什么，为什么必须进入首层，成功与失败边界分别落在哪里。**

本稿不是验收口径稿，不定义通过/不通过规则，不进入实现方案、接口设计、模块拆分和技术路线。

## 使用方式

- 本稿是《首层场景筛选稿》的逐项展开，不替代筛选稿
- 本稿用于帮助评审人、记录人和证据准备人理解每个场景的证明责任
- 本稿只压实业务语义和证明边界，不把场景写成测试步骤
- 后续若继续推进，应在本稿基础上形成“场景-成立条件映射”和“验收口径”

## 真实业务语义来源说明

为避免场景语义停留在抽象表述层，本稿参考当前仓库中已经存在的真实业务命令语义作为锚点，包括但不限于：

- 连接与状态：`connect`、`disconnect`、`status`
- 运动与停机：`home`、`stop-all`、`estop`
- 点胶链路：`dispenser start|pause|resume|stop`、`supply open|close`
- 设计输入与执行：`dxf-plan`、`dxf-dispense`
- 配方与版本：`recipe compare`、`recipe activate`

这些语义锚点只用于约束“场景在业务上像什么”，不意味着当前阶段进入命令级设计或实现约束。

## 一、S1 主干成功闭环

- 场景名称：`主干成功闭环`
- 代表语义：已连接、已回零、已激活配方、DXF 有效，完成“路径规划 → 点胶执行 → 结果输出”主干闭环
- 业务前提：存在有效设计输入、有效配方版本和可进入执行态的基本设备上下文
- 场景意图：证明工程替身系统不是只能承接局部观察、局部注入或局部状态切换，而是能把一条主干研发判断链真正闭合
- 证明责任：证明研发可以在脱离真机的条件下，把“设计输入是否可被理解、执行链是否可被进入、结果是否可被输出与判断”连成一条完整链路
- 关键关注点：不是单看任务是否启动，而是看路径规划、执行进入、执行推进和结果输出是否前后连贯且结论明确
- 成功边界：主干链条完整闭合，输出结果可观察、可比较、可解释，研发无需回真机才能完成主干判断
- 失败边界：只能跑通某个局部片段、无法输出有效结果、或结论仍依赖人工脑补链路是否闭合
- 纳入理由：若 `S1` 不能成立，则整个系统会退化为若干局部替身能力的集合，而不是可承接研发主链的工程替身系统

## 二、S2 前置条件阻断与有效反馈

- 场景名称：`前置条件阻断与有效反馈`
- 代表语义：未激活配方或未选择有效版本时，尝试启动 DXF 点胶任务被阻断，并明确说明阻断原因
- 业务前提：执行请求已经产生，但配方或版本前提不成立
- 场景意图：证明系统不只会在前提齐备时跑通主干，也能对“为什么此时不能开始”给出有效判断
- 证明责任：证明系统能把“配方/版本未就绪”从隐性依赖变成显性反馈，让研发在脱机环境下完成前置条件确认
- 关键关注点：阻断必须发生在正确位置，且反馈必须足够具体，能支撑研发判断卡点而不是只返回笼统失败
- 成功边界：任务被明确阻断，原因可定位到配方或版本有效性问题，研发能据此继续修正而不是转回真机试错
- 失败边界：任务误启动、阻断原因含混、或只能看见“失败”而不能判断失败责任归属
- 纳入理由：若 `S2` 缺失，团队会很容易把首层环境误当成“只能跑 happy path 的演示环境”

## 三、S3 前置条件阻断与有效反馈

- 场景名称：`前置条件阻断与有效反馈`
- 代表语义：未回零、供胶未开或关键设备状态未就绪时，启动任务被拒绝，并给出具体卡点
- 业务前提：执行请求存在，但运动、供胶或关键就绪态不满足进入条件
- 场景意图：证明系统能承接设备就绪判断，而不是只承接设计输入和业务命令表面流转
- 证明责任：证明“回零、供胶、就绪态”这类执行前提可以在工程替身系统内被表达、被判断、被拒绝，并形成有效反馈
- 关键关注点：要看系统是否能区分不同未就绪来源，而不是把所有不能执行的情况都压成同一种失败现象
- 成功边界：任务被正确拒绝，卡点落在明确的设备前提上，研发可据此继续定位联调问题
- 失败边界：系统无法表达设备前提、无法区分卡点来源、或明明未就绪却允许进入执行
- 纳入理由：若 `S3` 缺失，脱机环境就无法支撑“执行前为什么不能开始”这一类高频研发判断

## 四、S4 前置条件阻断与有效反馈

- 场景名称：`前置条件阻断与有效反馈`
- 代表语义：DXF 无效、空轨迹或关键轨迹参数无效时，规划或执行前被阻断并给出原因
- 业务前提：设计输入已经进入系统，但输入本身不足以形成有效执行对象
- 场景意图：证明工程替身系统承接的不只是执行阶段，还包括“设计输入是否足以进入执行链”的前置判断
- 证明责任：证明系统能够对设计输入问题形成显性判断，把“图纸/轨迹/参数是否有效”前移到脱机环境中处理
- 关键关注点：要看阻断是否发生在规划或执行前的合适位置，以及系统是否能把问题归到输入有效性而不是执行失败
- 成功边界：无效输入在进入执行前被识别并阻断，原因可被研发使用来修改设计输入或参数组合
- 失败边界：无效输入被带入后续流程才暴露、问题归因模糊、或只能依赖真机联调判断输入是否有意义
- 纳入理由：若 `S4` 缺失，“设计输入 → 路径生成”这段关键研发判断仍会留在真机侧

## 五、S5 关键操作链路切换

- 场景名称：`关键操作链路切换`
- 代表语义：主干任务执行中发生“暂停 → 继续 → 停止/复位 → 再进入”，状态流转与再进入行为正确
- 业务前提：系统已进入执行态，并需要处理中途操作链路切换
- 场景意图：证明首层环境不是只适合“一次启动跑到底”，而是能承接真实研发中频繁出现的操作链路变化
- 证明责任：证明暂停、继续、停止、复位和再进入之间的时序关系、状态切换和后续行为是可表达、可判断的
- 关键关注点：不是单看每个动作是否被接受，而是看动作前后状态是否一致、再进入条件是否清楚、链路是否可恢复
- 成功边界：关键操作切换后系统行为可预测、状态可解释、再进入路径清楚，研发可据此判断控制链路是否成立
- 失败边界：操作切换只表现为表面命令响应，内部状态失真、再进入条件不清，或后续行为不可判定
- 纳入理由：若 `S5` 缺失，首层环境会退化成单一路径执行器，无法支撑真实研发中的控制链路判断

## 六、S6 关键交互时序异常

- 场景名称：`关键交互时序异常`
- 代表语义：至少一个跨模块或跨接口边界上出现超时、错序或关键响应缺失，任务进入明确可判定状态
- 业务前提：存在真实跨边界交互，且该交互的时序关系会影响任务能否继续推进
- 场景意图：证明工程替身系统能前移暴露集成问题，而不是只验证单模块内部逻辑
- 证明责任：证明关键交互异常可以被主动构造，并在脱机环境下形成明确状态变化、错误归属或任务终止判断
- 关键关注点：要看异常是否落在真实交互边界上，以及系统是否能把“超时、错序、缺响应”转换成可判定结果
- 成功边界：交互异常被真实承接，系统进入明确状态，研发能判断问题来自交互时序而不是局部业务逻辑
- 失败边界：异常只能靠口头假设、无法被构造、无法形成明确状态，或最终又需要回真机判断集成责任
- 纳入理由：若 `S6` 缺失，“前移暴露集成问题”这一核心目标基本无法成立

## 七、S7 高频或高风险异常与恢复

- 场景名称：`高频或高风险异常与恢复`
- 代表语义：点胶阀、供胶链路或关键传感信号异常触发报警，经过处理后恢复到正确状态或安全停止
- 业务前提：系统处于执行中或即将执行状态，且关键执行链路发生高频或高风险异常
- 场景意图：证明系统不是只会处理主干成功链，也能承接典型异常、报警和恢复判断
- 证明责任：证明高频或高风险异常可以在工程替身环境中被构造、被识别、被处理，并产生恢复或安全停止结论
- 关键关注点：既要看异常本身是否被识别，也要看处理后系统是回到正确状态、进入安全停止，还是留在含糊不清的中间态
- 成功边界：异常进入明确报警或异常态，经处理后可恢复到合理状态，或明确进入安全停止并结束判断
- 失败边界：异常只表现为现象噪声、无法触发明确处理链、或恢复后的状态不可判断
- 纳入理由：若 `S7` 缺失，首层环境仍只能承接正常流，无法覆盖研发中最常见的异常联调判断

## 八、S8 高频或高风险异常与恢复

- 场景名称：`高频或高风险异常与恢复`
- 代表语义：运行中急停后进入锁定状态，随后完成复位、重新入场或明确终止判定
- 业务前提：任务已在运行过程中，且出现需要立即中断的高风险事件
- 场景意图：证明首层环境能承接高风险中断链路，而不是把最关键的安全相关判断全部留给真机
- 证明责任：证明急停触发、锁定进入、复位条件、重新入场或终止判定这些关键节点可以在脱机环境下形成清晰判断
- 关键关注点：要看急停后的锁定是否明确、后续是否允许直接继续、复位与再进入逻辑是否可解释
- 成功边界：急停后状态明确，后续路径只允许进入受控的复位、重新入场或终止分支，研发可做出明确结论
- 失败边界：急停仅表现为一次表面停止，后续状态不清、恢复链混乱、或安全相关判断仍必须依赖真机
- 纳入理由：若 `S8` 缺失，首层环境无法证明自己承接了高风险中断路径，系统成立性会明显不足

## 九、S9 关键数据组合对照

- 场景名称：`关键数据组合对照`
- 代表语义：同一 DXF/任务在两组关键配方版本、关键参数或关键配置下执行，对结果差异进行可比、可判定对照
- 业务前提：存在同一业务任务在不同数据组合下的可重复输入条件
- 场景意图：证明工程替身系统能承接真实的数据化研发判断，而不是只能在单一默认配置下跑一次流程
- 证明责任：证明系统能在相同任务框架下，稳定比较不同配方版本、关键参数或关键配置带来的结果差异
- 关键关注点：不是只看两次执行都完成，而是看差异是否可归因、可比较、可形成明确判断
- 成功边界：同一任务在不同数据组合下形成稳定对照，差异可观察、可解释、可判定，研发可据此完成版本或参数判断
- 失败边界：只能跑单一默认值、差异不可比、结果不可重演，或无法判断差异来自数据还是环境噪声
- 纳入理由：若 `S9` 缺失，工程替身系统就难以承接“版本/参数变化会怎样影响行为”这一真实研发判断

## 十、本稿结论

当前可冻结的结论是：

- `S1-S9` 并不是九个并列功能点，而是九个首层证明责任载体
- 每个场景都必须被理解为“在证明什么”，而不能只被理解为“要不要跑一下”
- 后续若进入“场景-成立条件映射”与“验收口径”，应以本稿为上位逐项说明依据

在当前阶段，首层业务场景逐项说明可冻结为 `Baseline v1.0` 的组成部分，并作为后续映射、验收和正式评审的直接输入。
