# Demo-1 Archive Path Fragmentation Root Cause Analysis

Date: 2026-04-27

Scope: HMI import of `D:\Projects\SiligenSuite\uploads\dxf\archive\Demo-1.dxf`

Mode: root-cause analysis only. This document records observed evidence, candidate root causes, and evidence gaps. It does not define a repair plan.

## 1. Symptom

After importing `uploads\dxf\archive\Demo-1.dxf` in HMI, the UI shows:

> 路径碎片化提示。路径较碎，当前结果已按例外规则放行，可继续预览与执行。这通常意味着 DXF 几何连通性较差，或导入顺序导致路径被拆成多段。

The displayed text is generated by HMI preview preflight when the current preview state has:

- `preview_diagnostic_code == "process_path_fragmentation"`
- `preview_validation_classification != "fail"`

Evidence:

- `modules/hmi-application/application/hmi_application/services/preview_preflight.py`
  - `current_preview_diagnostic_notice()` returns title `路径碎片化提示` and the matching detail when `preview_diagnostic_code` equals `process_path_fragmentation`.

## 2. Trigger Conditions

The code path that produces the HMI notice is:

1. Process path topology analysis detects path fragmentation.
2. Planning prepare stores `preview_diagnostic_code = "process_path_fragmentation"`.
3. Runtime preview snapshot returns that diagnostic metadata.
4. HMI renders the fragmentation notice from preview state.

Evidence:

- `modules/dispense-packaging/application/usecases/dispensing/PlanningUseCase.cpp`
  - `prepared.discontinuity_count = process_path_result.normalized.report.discontinuity_count`
  - `prepared.preview_diagnostic_code = process_path_result.topology_diagnostics.fragmentation_suspected ? "process_path_fragmentation" : ""`
- `modules/process-path/domain/trajectory/domain-services/TopologyRepairService.cpp`
  - `fragmentation_suspected` is true when `discontinuity_after > 0`, or when discontinuity existed before repair and repair was not applied.
- `modules/process-path/domain/trajectory/domain-services/GeometryNormalizer.cpp`
  - adjacent segment gaps above `continuity_tolerance` increment `discontinuity_count`.

## 3. Primary Hypothesis

The most likely root cause is that the archive DXF file itself is highly discontinuous in entity order. The current process-path chain interprets that discontinuity as path fragmentation, which surfaces as `process_path_fragmentation` in HMI.

Confidence: high.

Read-only DXF evidence for `uploads\dxf\archive\Demo-1.dxf`:

- File hash: `4AEF24148A89539B3AE6C27EB7F699F532FFB76D7E5C12B5CFE0008E85247C8E`
- File size: `76070` bytes
- Entity counts:
  - `LINE`: `388`
  - `POINT`: `3`
- Adjacent `LINE` continuity by raw DXF order:
  - Checked adjacent `LINE` end-to-next-start gaps using a `0.001mm` threshold.
  - `387 / 387` adjacent pairs exceeded the threshold.
  - Maximum observed adjacent gap: about `238.98mm`.
  - Early adjacent gaps were already in the `6mm` to `8mm` range.

This matches the user-facing diagnostic text: "DXF geometry connectivity is poor, or import order causes the path to be split into multiple sections."

## 4. Secondary Hypotheses

### 4.1 Preview Metadata And Production Gate Semantics Are Mixed In The UI Text

Confidence: medium-high.

The HMI notice says the result is allowed by exception and can continue preview/execution. Current contract text and runtime code distinguish that preview diagnostic metadata from production path-quality authority:

- `preview_diagnostic_code` remains preview metadata for HMI/CLI.
- `path_quality` is the production path-quality authority.
- Blocking `path_quality.reason_codes` must fail closed before runtime start.

Evidence:

- `shared/contracts/application/mappings/protocol-mapping.md`
  - `preview_*` fields are metadata and must not replace `path_quality` as production gate authority.
  - `path_quality.reason_codes` includes `process_path_fragmentation` and `path_discontinuity`.
  - Blocking reasons must block `dxf.job.start`.
- `docs/architecture/topics/dxf/dxf-motion-execution-contract-v1.md`
  - `path_quality` is the only production path-quality gate.
  - Any blocking reason must make `dxf.job.start` fail closed.
- `modules/runtime-execution/application/usecases/dispensing/DispensingWorkflowUseCase.cpp`
  - `RefreshPathQualityAssessment()` adds `process_path_fragmentation` and `path_discontinuity`.
  - `ResolvePathQualityGateFailure()` rejects blocking path quality before runtime start.

Interpretation: the HMI text explains why preview can be shown, but it should not be treated as proof that production start will be allowed.

### 4.2 Same File Name Does Not Mean Same Artifact

Confidence: medium.

The user imported `uploads\dxf\archive\Demo-1.dxf`, but the repository also contains `samples\dxf\Demo-1.dxf`. They are not identical:

- `uploads\dxf\archive\Demo-1.dxf`
  - SHA256: `4AEF24148A89539B3AE6C27EB7F699F532FFB76D7E5C12B5CFE0008E85247C8E`
- `samples\dxf\Demo-1.dxf`
  - SHA256: `6BE11B43F24E66271E6A08ECA764CFD41D5766180D2BC73CE9398735573D0948`

Interpretation: prior conclusions about `samples\dxf\Demo-1.dxf` cannot be applied directly to the archive file without current evidence from the archive file.

Likely impact:

## 8. Current Most Likely Root Cause

The most likely root cause is that `uploads\dxf\archive\Demo-1.dxf` is highly fragmented or discontinuous in its raw DXF entity order. The process-path chain detects that as topology fragmentation/discontinuity, emits `process_path_fragmentation`, and HMI renders the corresponding path fragmentation notice.

This conclusion is evidence-backed for the archive file's raw entity order and for the code path that maps fragmentation diagnostics to the HMI message. It remains incomplete for the exact live runtime payload until the current HMI/gateway response is captured.
