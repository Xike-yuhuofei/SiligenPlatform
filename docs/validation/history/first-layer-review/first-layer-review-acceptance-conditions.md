# 成立条件稿

## 文档定位

本稿用于把“工程替身系统在定义上何时算成立”压成一组门槛式条件。
这些条件不是能力愿望清单，也不是实现清单，而是判断这套系统是否配得上“工程替身系统”这个定义的成立门槛。

判断原则是：若某项条件缺失，会直接削弱以下至少一项能力，则该系统在定义上尚不能成立：

- 让关键判断尽可能脱离真机完成
- 把问题暴露前移到上机前
- 形成可重复、可比较、可判定的最小研发闭环

## 一、成立条件

### 1. 必须能够主动构造研发所需的验证上下文

系统必须支持研发主动进入有效验证上下文，而不是继续依赖真机自然跑到某个状态后再开始判断。

这里的“验证上下文”至少应覆盖：

- 前置条件成立与不成立的状态
- 可切入的关键流程阶段
- 用于验证判断的关键环境置位
- 进入判断所需的基本输入条件

### 2. 必须能够提供足以支撑研发判断的有效反馈

系统必须提供足以支撑研发判断的反馈，而不是只给出现象层结果。

这里的“有效反馈”至少应支持回答：

- 当前为什么能继续
- 当前为什么不能继续
- 问题卡在什么位置
- 当前结论是否成立

### 3. 必须能够验证关键交互时序、控制链路与操作序列

系统必须承接关键交互时序、控制链路与操作序列的验证责任，而不是只支持局部状态观察或单点逻辑判断。

若缺失这一条件，系统将退化为只适合做局部逻辑检查，无法承担真实联调与控制链验证。

### 4. 必须能够主动构造异常并检验恢复路径

系统必须允许研发主动构造异常，并判断异常后的处理、恢复或终止路径是否成立。

若只能验证正常流程，而不能验证异常路径与恢复路径，则系统在定义上不能承接研发流程中的关键风险判断。

### 5. 必须能够在数据化场景组合下确认行为

系统必须允许在不同关键参数、配方版本、配置组合或输入条件下形成可比较结论，而不是只能跑单一默认路径。

若缺失这一条件，系统将无法承担真实研发中的数据对照、变更确认和差异判断责任。

### 6. 必须能够稳定重演、结果可比较且结论可判定

系统必须支持同一验证场景稳定重复执行，并使结果可比较、结论可判定。

若验证结果依赖一次性现象、现场经验或临场解释，则系统不能构成工程级的研发替身环境。

## 二、使用约束

本稿中的六项条件应按“同时成立”理解，而不是按“任选其一”理解。

这意味着：

- 缺失任一条件，都不能判定系统在定义上成立
- 某些条件可以在首层范围内只由少量高判别力场景证明，但不能被完全省略
- 后续范围裁剪只能裁剪覆盖深度，不能裁掉成立条件本身

## 三、与其他文稿的关系

- 《工程替身系统总纲说明页》给出上位定义和价值标准
- 《需求定义稿》说明为什么存在、必须承接什么、明确不承接什么
- 本稿把“必须承接什么”压成可用于判定成立与否的门槛式条件
- 《非目标边界》负责排除不应进入当前成立条件的方向

若后续场景、范围或评审口径不能映射回本稿六项成立条件，则说明该内容尚不能进入当前系统的成立基线。

## 四、本稿结论

当前可冻结的成立结论是：

一个面向研发流程的工程替身系统，若不能同时做到以下六件事，就尚不能在定义上成立：

1. 主动构造验证上下文
2. 提供有效反馈
3. 验证关键交互时序、控制链路与操作序列
4. 主动构造异常并检验恢复路径
5. 在数据化场景组合下确认行为
6. 稳定重演、结果可比较且结论可判定
