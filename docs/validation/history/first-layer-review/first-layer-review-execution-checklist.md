# 正式评审执行清单

## 冻结信息

- 评审包名称：`首层成立性评审包`
- 基线版本：`Baseline v1.0`
- 适用范围：`S1-S9 首层证明场景`
- 使用原则：`本轮评审唯一口径，评审中不得并行修改定义`
- 当前版状态：`可直接投入正式评审`

## 评审主题

围绕 `S1-S9` 首层证明场景，判断首层最小研发闭环是否真实成立。

## 禁止项

- 不讨论实现方案
- 不讨论技术选型
- 不讨论排期与资源
- 不用低判别力场景替代当前首层证明场景
- 不把出现现象误判为完成证明责任

## 下一版轻量增强项

- 会前阅读要求统一标成“必须读 / 应读 / 核心评审人必须读”
- 单场景评审固定为“四步闭环”
- 会后补一份“阻断结论总表”

## 会议执行顺序

### 单场景四步闭环

1. 场景说明  
   说明场景前提、证明责任、成功/失败边界、本次证据输入。
2. 范围裁剪  
   用《核心范围分层稿》判断该场景承担的证明责任是否仍属首层必需、是否存在被弱场景替代的风险。
3. 验收判定  
   结合《成立条件稿》《核心范围分层稿》《验收口径稿》与单场景评审记录模板，判断关键判断是否成立、是否可比较、是否可判定、是否稳定重演、是否补上首层缺口。
4. 填表记录  
   立即填写该场景的通用首页表单和专项附页。

## 会中判定规则

- 只看证明责任，不看功能多少
- 只看是否补上首层缺口，不看实现是否优雅
- 不能用更容易做的弱场景替代现有场景
- 不能把现象当结论
- 不能把一次跑通当稳定重演
- 不能因实现困难而降低验收口径

### 结论级别

- `通过`：证明责任成立，且无阻断性证据缺口
- `不通过`：证明责任未成立
- `有条件通过`：主体成立，但边界缺口未清
- `待补证据`：证据不够，暂不下判断

### 结论使用约束

- 若“是否补上首层缺口”不成立，不能判通过
- 若“是否稳定重演”不成立，通常不能判通过
- 若仍依赖人工解释完成结论判断，通常不能判通过

## 首层成立判定规则

### 可判定“首层成立”

- 阻断项已清零
- `S1-S9` 各自承担的首层证明责任已被覆盖
- 不存在必须回真机才能完成的关键判断缺口
- 已形成可比较、可判定、可重演的最小闭环

### 不可判定“首层成立”

- 仍存在阻断项
- 某类首层缺口无人覆盖
- 关键场景虽有现象，但未完成证明责任
- 回归入口或集成前移能力仍未成立

固定原则：若仍有阻断项，只允许围绕缺口重审，不允许回到抽象定义层重开问题空间。
