# 验收口径稿

## 文档定位

本稿基于《成立条件稿》《首层业务场景逐项说明稿》和《场景-成立条件映射稿》，只回答一个问题：

**首层正式评审时，`S1-S9` 应该按什么口径判定“证明责任成立或未成立”。**

本稿不是实现方案，不定义接口和技术路线，也不把评审过程展开成测试步骤脚本。

## 使用方式

- 本稿用于统一单场景评审中的“通过 / 不通过 / 有条件通过 / 待补证据”使用边界
- 本稿用于统一整体结论中的“首层成立 / 首层不成立 / 待补证据后复审”使用边界
- 本稿是《首层业务场景逐项说明稿》和《场景-成立条件映射稿》的判定展开，不替代这两份文稿
- 单场景记录表、结论汇总表和阻断总表的判定用语，均应以本稿为准

## 一、验收总原则

### 1. 只判证明责任，不判功能多少

- 单场景评审要判断的是该场景承担的证明责任是否成立
- 不以功能列表长短、界面多少、实现是否完整替代证明责任判断

### 2. 只判关键判断是否脱离真机完成，不判“像不像真机”

- 场景是否成立，应回到“研发关键判断是否已可在脱机环境中完成”
- 不允许用展示效果、还原度或表现完整度替代成立判断

### 3. 只判可判定闭环，不判一次性现象

- 若某场景只出现一次现象，但不能形成稳定可比较结论，则不能按成立处理
- 单次跑通不自动等于证明责任成立

### 4. 只判当前场景承担的边界，不做越界加分

- 某场景只需证明自己承担的那部分责任
- 不允许因为附带展示了额外能力，就放宽其本应承担的核心判定要求

## 二、单场景判定前提

单场景进入正式判定前，至少应具备以下输入：

- 业务前提明确
- 验证上下文明确
- 关键输入条件明确
- 判定基线明确
- 证据来源明确

若以上任一项缺失，通常只能判为 `待补证据`，不应直接下 `通过` 或 `不通过`。

## 三、单场景必判项

每个 `S1-S9` 场景正式判定时，至少应同时检查以下六项：

1. 该场景承担的证明责任是否被真实触发  
   不能只出现外围现象，必须真正进入该场景应承担的判断核心。

2. 该场景对应的成立条件是否被覆盖  
   至少要能映回《场景-成立条件映射稿》中列出的核心覆盖条件。

3. 结果是否足以支撑研发判断  
   不能只有现象记录，必须能回答“当前为什么成立 / 为什么不成立”。

4. 是否补上该场景对应的首层缺口  
   若场景跑过了，但关键缺口仍未被补上，则不能判 `通过`。

5. 结果是否可比较、可判定  
   不能只依赖口头解释或临场经验得出结论。

6. 是否具备稳定重演基础  
   若结果明显依赖偶然性，通常不能判 `通过`。

## 四、单场景结论级别

### 1. `通过`

仅当以下条件同时满足时，单场景才可判为 `通过`：

- 场景承担的证明责任已成立
- 对应核心成立条件已被覆盖
- 该场景对应的首层缺口已被补上
- 结果可比较、可判定
- 不存在阻断性证据缺口

说明：

- `通过` 只表示该场景承担的证明责任成立
- `通过` 不表示该类问题已广覆盖，也不表示后续无需扩展

### 2. `不通过`

出现以下任一情形，通常应判为 `不通过`：

- 场景承担的证明责任未成立
- 关键成立条件未被实际覆盖
- 场景虽有现象，但未补上对应首层缺口
- 结果表面存在，但无法支撑明确研发判断

说明：

- `不通过` 是“责任未成立”，不是“体验不好”或“实现不优雅”

### 3. `有条件通过`

仅在以下情形可使用：

- 场景主体证明责任已大体成立
- 但边界缺口、证据收口或复核项仍未完全清理
- 这些遗留问题尚未达到阻断首层成立的程度

说明：

- `有条件通过` 不是宽松版 `通过`
- 若遗留问题已经影响是否补上首层缺口，不能判 `有条件通过`

### 4. `待补证据`

出现以下情形，通常应判为 `待补证据`：

- 判定基线缺失
- 证据来源缺失
- 结论需要依赖大量口头补充才能成立
- 当前无法明确判断是“责任未成立”还是“证据不足”

说明：

- `待补证据` 不是“默认先过”
- 在 `待补证据` 状态下，整体结论通常不能直接判为 `首层成立`

## 五、阻断项判定口径

以下类型问题，通常应进入阻断判断：

- 某类成立条件在首层证明集中事实上无人可证
- 某关键场景虽被执行，但其核心证明责任未成立
- 某关键判断仍必须回真机才能完成
- 某关键结论无法形成可比较、可判定、可重演闭环

若某问题满足以上任一条，且会直接影响“首层成立”判断，则应进入 [阻断结论总表](./first-layer-review-blocker-matrix.md)。

## 六、整体结论口径

### 1. `首层成立`

仅当以下条件同时满足时，整体才可判为 `首层成立`：

- `S1-S9` 各自承担的首层证明责任均已有结论承接
- 不存在未解除的阻断项
- 不存在必须回真机才能完成的关键判断缺口
- 已形成可比较、可判定、可重演的最小研发闭环

### 2. `首层不成立`

出现以下任一情形，通常应判为 `首层不成立`：

- 某类首层证明责任无人承担
- 关键场景未完成其核心证明责任
- 阻断项仍然存在且未解除
- 最小研发闭环尚未形成

### 3. `待补证据后复审`

适用于以下情况：

- 主要证明责任框架已建立
- 但仍有若干场景停留在 `待补证据`
- 当前不足以下 `首层成立` 或 `首层不成立` 的最终判断

说明：

- 该结论的本质是“暂缓整体结论”，不是“默认趋向通过”

## 七、口径使用约束

- 不允许用“弱场景跑通”替代“首层必需场景成立”
- 不允许把“现象出现”直接写成“证明责任成立”
- 不允许因实现困难下调验收口径
- 不允许把 `有条件通过` 当作回避明确判断的缓冲区
- 不允许跳过《场景-成立条件映射稿》直接生成无法回映到 `C1-C6` 的验收结论

## 八、本稿结论

当前可冻结的结论是：

- 首层正式评审必须围绕“证明责任是否成立”来判，而不是围绕功能多少来判
- 单场景结论与整体结论都必须以“是否补上首层缺口、是否形成可判定闭环”为核心口径
- `通过 / 不通过 / 有条件通过 / 待补证据` 与 `首层成立 / 首层不成立 / 待补证据后复审` 的使用边界已可冻结

在当前阶段，《验收口径稿》可纳入 `Baseline v1.0`，并作为正式评审时单场景判定、结论汇总和阻断判断的直接依据。
