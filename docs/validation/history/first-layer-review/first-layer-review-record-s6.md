## 一、场景说明

- 业务前提：应存在至少一个真实跨模块或跨接口边界，并可构造超时、错序或响应缺失
- 验证上下文：系统已进入需要等待、接收或依赖对端响应的阶段
- 关键输入条件：应明确异常注入点、异常类型与预期系统反应
- 证明责任摘要：证明集成问题能够在首层被主动构造并形成明确可判定结论
- 成功边界：异常触发后任务进入明确定义状态，系统反馈可用于定位问题
- 失败边界：异常无感、状态悬空、或只能凭人工经验判断发生了什么

## 二、范围裁剪

- 是否仍属首层必需：以《核心范围分层稿》为准；当前默认应为“是”
- 是否可被其他场景覆盖：通常不应被本地正常流程场景替代
- 是否存在弱场景替代风险：重点防止用本地 stub 成功路径替代真实时序异常验证
- 裁剪结论：仍属首层必需，本轮不得用本地正常路径替代真实时序异常。

## 三、验收判定

- 判定基线：异常被正确识别，系统进入可比较、可判定状态，且结论能前移暴露集成问题
- 证据来源：建议包含请求/响应记录、时序日志、超时或错序证据与状态结果
- 是否涉及真实跨边界交互：应重点核对
- 是否完成重复验证：建议重点核对
- 六项必判项：
  - 证明责任是否被真实触发：
  - 核心成立条件是否被覆盖：
  - 结果是否足以支撑研发判断：
  - 是否补上首层缺口：
  - 是否可比较、可判定：
  - 是否具备稳定重演基础：
- 是否补上首层缺口：应对应“集成前移暴露缺口”
