# 需求定义稿

## 文档定位

本稿是阶段 1“需求定义冻结”的主文稿。
它服务于一件事：把“工程替身系统”在定义层面的需求边界先冻结下来，使后续的非目标边界、成立条件、范围裁剪和评审口径都有明确上位依据。

本稿不回答以下问题：

- 不回答怎么实现
- 不回答如何做模块拆分
- 不回答采用什么技术路线
- 不回答接口、部署、排期和资源

## 一、为什么存在

这套系统存在的根本原因，不是为了构建一个更像设备的仿真软件，而是为了在研发流程中建立一套可替代真机承担关键判断责任的工程环境。

它要解决的核心问题是：

- 研发过程中的关键判断长期依赖真机，导致开发推进受设备条件约束
- 跨模块、跨接口、跨时序的问题往往在上机后才暴露，反馈链路过长
- 很多验证结论只能依赖现场操作或经验判断，难以重复、难以比较、难以稳定复现
- 自动化验证和回归判断缺乏稳定载体，导致问题复现和变更确认成本过高

因此，这套系统存在的意义，在于让研发活动中的关键判断尽可能在脱离真机条件下完成，并形成可重复、可比较、可判定的最小研发闭环。

## 二、对象定义

当前要定义的对象，是一套面向研发流程的**工程替身系统**。

它不是：

- 展示型设备仿真产品
- 培训型操作演示系统
- 数字孪生系统
- 真机物理验收替代物
- 对既有仿真系统的重写版

它是：

- 面向开发的逻辑验证环境
- 面向联调的跨边界确认环境
- 面向验证的可判定结论形成环境
- 面向异常确认的问题定位与恢复判断环境
- 面向回归判断的稳定重演与结果比较环境

## 三、需求核心

从定义层看，这套系统必须承接的不是“功能数量”，而是研发流程中的关键判断责任。

在当前阶段，需求核心应固定为以下几类：

### 1. 研发上下文进入能力

系统必须支持研发进入原本依赖真机才能进入的有效验证上下文，而不能继续依赖现场把设备自然跑到某个状态后再开始判断。

### 2. 有效反馈形成能力

系统必须提供足以支撑研发判断的反馈，而不是只给出“有现象”或“跑起来了”的表层结果。
反馈必须能用于说明当前为什么能继续、为什么不能继续、问题卡在什么位置、当前结论是否成立。

### 3. 关键链路验证能力

系统必须承接研发对关键交互时序、控制链路和操作序列的验证责任，而不是只支持局部逻辑或单点状态观察。

### 4. 异常与恢复判断能力

系统必须允许研发主动构造异常，并判断异常后的处理、恢复或终止路径是否成立，而不是只承接正常流程。

### 5. 数据化场景对照能力

系统必须允许在不同关键参数、配方版本、配置组合或输入条件下形成可比较结论，而不是只能运行单一默认路径。

### 6. 稳定重演能力

系统必须支持同一验证场景的稳定重复执行，使结论能够回溯、比较和复查，而不是依赖一次性现象或临场解释。

## 四、明确不承接什么

在定义层面，当前不应把以下内容写入这套系统的核心需求：

- 不以外观、三维画面、空间形态或展示效果作为目标
- 不以“尽可能完整复制设备世界”作为默认方向
- 不以替代真机做最终物理验收、工艺验收或最终性能确认作为目标
- 不以培训、演示、对外展示为当前建设目标
- 不以现有在研仿真系统的能力边界、模块结构或实现习惯反推需求

这些内容即使未来可能有价值，也不应在当前阶段被写成定义成立所必须承接的需求。

## 五、成立判断

这套系统在定义上是否成立，不取决于它做了多少功能，也不取决于它是否更像设备，而取决于以下总判断是否成立：

- 是否已经能够承接研发流程中的关键判断责任
- 是否已经让关键判断尽可能脱离真机完成
- 是否已经形成可重复、可比较、可判定的最小研发闭环
- 是否已经具备支撑后续联调、异常确认和回归判断的基础

如果上述判断不能成立，那么即使系统具备较多表现层能力，也不能被视为一个成立的工程替身系统。

## 六、对后续文稿的约束

本稿冻结后，后续文稿应按以下顺序继续收束：

1. 把“明确不承接什么”单独压实为《非目标边界》
2. 把“成立判断”压实为《成立条件稿》
3. 再进入范围边界、首层分层、场景证明集与评审口径

后续任何文稿若与本稿冲突，应以本稿为准进行修正，而不是反过来修改本稿去适应实现便利性。

## 七、本稿结论

当前可以冻结的定义结论是：

这套系统应被定义为一套面向研发流程的工程替身系统。
它存在的意义，是让研发活动中的关键判断尽可能在脱离真机条件下完成，并形成可重复、可比较、可判定的最小研发闭环。
因此，本阶段的需求定义重点，不是扩展功能愿望，而是先冻结这套系统必须承接的判断责任、明确不承接的方向，以及判断其是否成立的定义标准。
