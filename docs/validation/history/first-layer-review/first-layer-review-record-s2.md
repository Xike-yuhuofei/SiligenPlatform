- 证明责任锚点：证明系统能判断“为什么不能开始”，而不是只会跑通 happy path

## 一、场景说明

- 业务前提：应刻意构造“配方未激活”或“版本无效”的前置缺失条件
- 验证上下文：系统处于可尝试启动任务但尚未满足数据前提的状态
- 关键输入条件：应明确本次缺失的是哪类配方前提、对应任务是什么
- 证明责任摘要：证明任务在数据前提不成立时会被正确阻断，且能给出可用于研发判断的反馈
- 成功边界：任务被拒绝，拒绝原因明确、可判定、可复现
- 失败边界：任务误启动、拒绝原因含糊、或只能通过人工解释理解卡点

## 二、范围裁剪

- 是否仍属首层必需：以《核心范围分层稿》为准；当前默认应为“是”
- 是否可被其他场景覆盖：仅能被同类“数据前置条件阻断”场景覆盖，通常不应删除
- 是否存在弱场景替代风险：重点防止用静态校验提示替代真实启动阻断
- 裁剪结论：仍属首层必需，本轮不得用静态提示替代真实阻断。

## 三、验收判定

- 判定基线：阻断发生在正确阶段，且反馈能直接支撑“为何不能开始”的判断
- 证据来源：建议包含启动请求、阻断反馈、状态记录与结论依据
- 是否涉及真实跨边界交互：按实际填写
- 是否完成重复验证：建议重点核对
- 六项必判项：
  - 证明责任是否被真实触发：
  - 核心成立条件是否被覆盖：
  - 结果是否足以支撑研发判断：
  - 是否补上首层缺口：
  - 是否可比较、可判定：
  - 是否具备稳定重演基础：
- 是否补上首层缺口：应对应“数据前置条件阻断缺口”
