## 场景定义

- 场景名称：`关键数据组合对照`
- 代表语义：同一 DXF/任务在两组关键配方版本、关键参数或关键配置下执行，对结果差异进行可比、可判定对照
- 证明责任锚点：证明系统具备真实的数据化场景承接能力，而不是只能跑单一默认配置
- 核心覆盖条件：`C1 C2 C5`
- 次级带出条件：`C6`

## 一、场景说明

- 业务前提：应准备同一任务在两组关键数据配置下的可比较输入
- 验证上下文：系统应能在同一验证框架下重复执行并输出可比结果
- 关键输入条件：应明确两组配方版本、关键参数或关键配置的差异点
- 证明责任摘要：证明数据化场景切换后，系统行为差异可以被稳定重演、可比较、可判定
- 成功边界：两组输入下结果差异明确、可解释、可回溯
- 失败边界：结果不可比、差异不可判定、或无法确认差异由关键数据组合引起

## 二、范围裁剪

- 是否仍属首层必需：以《核心范围分层稿》为准；当前默认应为“是”
- 是否可被其他场景覆盖：通常不应被单一默认配置运行场景替代
- 是否存在弱场景替代风险：重点防止只做一次运行结果展示而不做成组对照
- 裁剪结论：仍属首层必需，本轮不得用单次运行展示替代成组对照。

## 三、验收判定

- 判定基线：同一任务在关键数据组合变化下能形成稳定、可比较、可判定的结果对照
- 证据来源：建议包含两组输入、对应输出、差异说明与对照依据
- 是否涉及真实跨边界交互：按实际填写
- 是否完成重复验证：应重点核对
- 六项必判项：
  - 证明责任是否被真实触发：
  - 核心成立条件是否被覆盖：
  - 结果是否足以支撑研发判断：
  - 是否补上首层缺口：
  - 是否可比较、可判定：
  - 是否具备稳定重演基础：
- 是否补上首层缺口：应对应“关键数据组合对照缺口”
