# 场景-成立条件映射稿

## 文档定位

本稿基于《成立条件稿》《首层场景筛选稿》和《首层业务场景逐项说明稿》，只回答一个问题：

**`S1-S9` 分别在覆盖哪些成立条件，整体证明集是否已经把 `C1-C6` 全部承接起来。**

本稿不是验收口径稿，不定义通过/不通过阈值，也不进入实现方案、接口设计和技术路线。

## 使用方式

- 本稿用于把“场景在证明什么”进一步压成“场景在覆盖哪些成立条件”
- 本稿用于检查首层证明集是否存在成立条件无人承接的空洞
- 本稿用于约束后续验收口径，不允许出现无法映回 `C1-C6` 的验收条目
- 本稿不替代《首层业务场景逐项说明稿》，两者应配套阅读

## 成立条件简写

- `C1`：主动构造验证上下文
- `C2`：提供有效反馈
- `C3`：验证关键交互时序、控制链路与操作序列
- `C4`：主动构造异常并检验恢复路径
- `C5`：在数据化场景组合下确认行为
- `C6`：稳定重演、结果可比较且结论可判定

## 二、按成立条件反看场景覆盖

### C1 主动构造验证上下文

- 主要承接场景：`S1-S9`
- 说明：首层九个场景全部要求研发主动进入某种可判定上下文，因此 `C1` 是整个证明集的共同底座
- 当前结论：`C1` 已被全体场景共同覆盖，不存在无人承接空洞

### C2 提供有效反馈

- 主要承接场景：`S1-S9`
- 说明：无论是主干成功、前置阻断、交互异常还是数据对照，若不能形成有效反馈，则这些场景都无法完成证明责任
- 当前结论：`C2` 同样由全体场景共同承接，是首层最普遍的成立条件

### C3 验证关键交互时序、控制链路与操作序列

- 主要承接场景：`S1`、`S3`、`S5`、`S6`、`S8`
- 说明：`C3` 不是所有场景的主责任，但必须至少由主链、执行前提、操作切换、交互异常和高风险中断这几类场景共同支撑
- 当前结论：`C3` 已被多类高判别力场景覆盖，不依赖单一场景成立

### C4 主动构造异常并检验恢复路径

- 主要承接场景：`S7`、`S8`
- 说明：`C4` 在首层不追求广覆盖，而是集中由“典型异常恢复”和“高风险中断恢复”两类场景承担
- 当前结论：`C4` 当前依赖 `S7` 与 `S8` 共同成立；若两者缺失其一，异常与恢复覆盖将明显变薄

### C5 在数据化场景组合下确认行为

- 主要承接场景：`S1`、`S2`、`S4`、`S9`
- 说明：`C5` 不仅体现在显性的双组数据对照，也体现在配方/版本、设计输入有效性和主干执行上下文中的关键数据条件变化
- 当前结论：`C5` 已有主场景与专项场景共同承接，其中 `S9` 是最直接的专项证明场景

### C6 稳定重演、结果可比较且结论可判定

- 主要承接场景：`S1-S9`
- 说明：`C6` 不属于某一个专项能力，而是全部首层场景共同成立时才真正被证明
- 当前结论：`C6` 是所有场景的共同收束条件，后续验收口径必须对其单独压实

## 三、首层证明集的映射结论

- `S1-S9` 组合后，已经在定义层把 `C1-C6` 全部纳入首层证明集
- 其中 `C1`、`C2`、`C6` 由全体场景共同承接，是整个证明集的底座条件
- `C3` 主要由主链、执行前提、操作切换、交互异常和高风险中断场景承担
- `C4` 主要由 `S7`、`S8` 两类异常与恢复场景承担
- `C5` 主要由 `S1`、`S2`、`S4`、`S9` 共同承担，其中 `S9` 是最直接的数据化对照场景

当前首层不存在“成立条件完全无人承接”的结构性空洞，但不同条件的承接密度并不相同。  
后续若进入验收口径阶段，应重点压实 `C4` 与 `C6`，避免它们只停留在说明性表述而未形成明确判定入口。

## 四、使用约束

- 不允许把某场景覆盖到的成立条件机械理解为“该条件只由此场景承担”
- 不允许因为某项成立条件已有一个场景覆盖，就删除同类高判别力场景
- 不允许在后续验收口径中新增无法映回 `C1-C6` 的判定项
- 不允许用实现便利性反向削弱本稿中的映射关系

## 五、本稿结论

当前可冻结的结论是：

- `S1-S9` 已经形成一套可回映到 `C1-C6` 的首层证明集
- 每个场景都不是孤立存在，而是在承担一组明确的成立条件覆盖责任
- 后续若继续推进验收口径和正式评审，应以本稿作为“场景为何能证明成立条件”的直接上位依据

在当前阶段，《场景-成立条件映射稿》可纳入 `Baseline v1.0`，并作为后续验收口径和评审执行的直接输入。
