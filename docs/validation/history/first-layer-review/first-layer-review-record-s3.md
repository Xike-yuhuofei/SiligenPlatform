## 一、场景说明

- 业务前提：应构造“未回零”“供胶未开”或关键设备状态未就绪的真实前置缺口
- 验证上下文：系统处于准备启动任务但设备就绪链未闭合的状态
- 关键输入条件：应明确缺失的是哪一个就绪条件，以及尝试启动的任务
- 证明责任摘要：证明设备状态与执行前置判断不是摆设，而是真正参与任务准入判定
- 成功边界：启动被拒绝，且反馈明确指出具体未就绪卡点
- 失败边界：误允许启动、反馈只有笼统失败、或无法区分具体卡点

## 二、范围裁剪

- 是否仍属首层必需：以《核心范围分层稿》为准；当前默认应为“是”
- 是否可被其他场景覆盖：通常不应被 `S2` 替代，因为它关注设备就绪而非数据前提
- 是否存在弱场景替代风险：重点防止用静态状态展示替代真实启动前置判断
- 裁剪结论：仍属首层必需，本轮不得用静态状态展示替代真实准入阻断。

## 三、验收判定

- 判定基线：设备未就绪时任务必被正确拦截，且卡点可直接用于研发判断
- 证据来源：建议包含设备状态快照、启动请求、拒绝反馈与状态流转记录
- 是否涉及真实跨边界交互：建议重点核对
- 是否完成重复验证：建议重点核对
- 六项必判项：

  - 核心成立条件是否被覆盖：
  - 结果是否足以支撑研发判断：
  - 是否补上首层缺口：

  - 是否具备稳定重演基础：
