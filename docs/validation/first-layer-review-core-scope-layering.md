# 核心范围分层稿

## 文档定位

本稿用于在《范围定义稿》已冻结的前提下，把“核心范围”进一步分成两层：

- `首层必需承接`
- `核心内后置展开`

本稿不改变《范围定义稿》的核心/扩展/真机保留三分边界，只回答一个问题：

在已经确认属于核心范围的内容里，哪些缺失就无法形成最小研发闭环，哪些仍属核心但可以在首层成立后继续展开。

## 一、分层前提

- 核心范围定义的是系统成立必须覆盖的边界，不等于首期版本必须一次性交付的全部内容
- 本稿的分层对象只限于《范围定义稿》已冻结的核心范围，不重新裁掉核心范围本身
- 本稿不进入场景筛选，不以具体业务场景数量决定分层结论
- 本稿不进入实现顺序、模块拆分、接口设计和技术路线

## 二、分层判断原则

### 1. 首层必需承接

若某项内容缺失，会直接导致以下至少一项不能成立，则应归入首层必需承接：

- 无法形成最小研发闭环
- 关键判断仍必须回到真机才能完成
- 问题无法前移暴露到上机前
- 结论无法做到可比较、可判定、可重演

### 2. 核心内后置展开

若某项内容仍明确服务于核心判断责任，但缺失并不会立刻否定系统成立，只会影响覆盖深度、验证效率或长期收益，则应归入核心内后置展开。

## 三、首层目标

首层目标不是覆盖所有核心能力的全部深度，而是先形成一个可落地、可判定、可重演的最小研发闭环，使工程替身系统在定义上真实成立，并为后续扩展提供稳定基座。

## 四、逐项分层

### 1. 研发验证上下文的主动构造

`首层必需承接`

- 能主动进入研发判断所需的基本前置条件成立/不成立状态
- 能切入关键流程阶段，而不是只能从自然起点顺跑
- 能为关键判断准备最基本的输入条件和环境置位

`核心内后置展开`

- 更细粒度的上下文组合与更多中间阶段切入
- 更高覆盖度的前置条件矩阵
- 更复杂的连续阶段跳转与复合上下文构造

### 2. 支撑研发判断的有效反馈

`首层必需承接`

- 能明确说明当前为什么能继续或不能继续
- 能明确给出当前卡点和当前结论是否成立
- 能输出足以支撑研发判断的关键状态与结果

`核心内后置展开`

- 更丰富的反馈维度与更强的观察便利性
- 更细的对比视角、历史信息与辅助分析信息
- 更高密度的诊断信息与更完整的结论辅助材料

### 3. 关键交互、控制链路与操作序列的验证

`首层必需承接`

- 至少承接一条主干控制链路的完整验证
- 能验证关键操作序列的正确流转
- 至少承接一类真实跨边界交互或关键链路判断

`核心内后置展开`

- 更多业务支路的交互链验证
- 更复杂的时序变化、错序和边界组合
- 更广覆盖的操作链路和更深层的控制分支

### 4. 异常构造与恢复路径判断

`首层必需承接`

- 能主动构造至少一类代表性异常
- 能判断异常后的处理、恢复或安全终止是否成立
- 能形成异常路径上的明确结论，而不是只停留在“有报警”

`核心内后置展开`

- 更多异常类型、异常组合和异常层级
- 更复杂的恢复链条和更长路径的状态回归
- 更高覆盖度的高频/高风险异常集合

### 5. 数据化场景组合下的行为确认

`首层必需承接`

- 能在同一任务下比较至少一组关键数据变化带来的行为差异
- 能把差异判断落到可比较、可判定的结论
- 能支撑最基本的数据化场景对照责任

`核心内后置展开`

- 更广的参数矩阵、配方版本组合与配置组合
- 更多差异来源的对照能力
- 更高密度的数据化回归样本覆盖

### 6. 稳定重演、比较与判定

`首层必需承接`

- 同一验证场景可重复执行
- 结果可比较，结论可判定
- 关键判断不依赖一次性现象或临场解释

`核心内后置展开`

- 更大规模的重演样本与回归样本集
- 更高频率的重复验证和更强的批量比较能力
- 更丰富的证据沉淀与长期结论复查能力

## 五、使用约束

- 本稿中的“首层必需承接”不等于“系统全部核心能力”
- 本稿中的“核心内后置展开”不等于“可移出核心范围”
- 后续场景筛选只能在本稿定义的首层必需承接内选择高判别力代表场景
- 若某项内容不属于首层必需承接，则不应被强行纳入首层证明集

## 六、本稿结论

当前可冻结的分层结论是：

《范围定义稿》中的核心范围，应进一步分成“首层必需承接”和“核心内后置展开”两层。
其中，能够支撑最小研发闭环成立、让关键判断尽可能脱离真机完成、并形成可比较可判定结果的那一层，属于首层必需承接；
仍服务核心判断责任、但缺失并不会立刻否定系统成立的那一层，属于核心内后置展开。

后续若进入首层场景筛选，应只围绕“首层必需承接”这一层继续收束。
