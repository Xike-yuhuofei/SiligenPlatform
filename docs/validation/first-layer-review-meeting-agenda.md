# 首层评审会议议程页

## 会议定位

- 会议名称：`正式首层范围评审 / 首层成立性评审`
- 基线版本：`Baseline v1.0`
- 评审对象：`S1-S9 首层证明场景`
- 会议目标：围绕 `S1-S9` 的证明责任，判断首层最小研发闭环是否真实成立
- 会议边界：只讨论首层证明责任是否成立，不讨论实现方案、技术选型、排期与资源

## 参会角色

- 主持人：控制会议节奏，确保不偏离评审边界
- 场景陈述人：逐项说明场景前提、证据输入和证明责任
- 裁剪把关人：判断该场景是否仍属首层必需、是否可被替代
- 记录人：同步填写评审记录、结论汇总和阻断总表
- 结论确认人：对单场景结论和整体结论做最终确认

## 推荐议程

### 1. 开场与边界确认

- 重申评审主题、会议边界和禁止项
- 确认参会角色是否到齐
- 确认记录模板、结论汇总表和阻断总表已打开

### 2. 会前进入条件复核

- 是否已接受“工程级真机替身”主定义句
- 是否已接受《成立条件稿》与《非目标边界》
- 是否已接受《范围定义稿》
- 是否已接受《核心范围分层稿》
- 是否已接受《首层场景筛选稿》
- 是否已接受《首层业务场景逐项说明稿》
- 是否已接受《场景-成立条件映射稿》
- 是否已接受《验收口径稿》
- 是否已接受《正式评审执行包冻结稿》
- 是否已接受《证据补齐与首轮复审准备稿》
- 是否已接受《逐场景证据采集与复审输入包组装稿》
- 是否已接受 `S1-S9` 为当前首层证明集
- 是否已接受“通过不等于广覆盖”
- 是否已接受会议不讨论实现方案、排期、资源

### 3. 证据准备状态确认

- 逐项确认 `S1-S9` 是否已具备判定基线
- 逐项确认 `S1-S9` 是否已具备证据来源
- 对“是否可开评 = 否”的场景先挂起，不进入正式判定

### 4. 单场景正式评审

每个场景固定按“四步闭环”进行：

1. 场景说明
2. 范围裁剪
3. 验收判定
4. 填表记录

推荐顺序：

- `S1`
- `S2`
- `S3`
- `S4`
- `S5`
- `S6`
- `S7`
- `S8`
- `S9`

### 5. 整体结论汇总

- 汇总已通过 / 不通过 / 有条件通过 / 待补证据场景
- 汇总已补上的首层缺口与尚未补上的首层缺口
- 逐项确认阻断项是否成立
- 判断整体是否可判为 `首层成立`

### 6. 会后动作确认

- 明确需补证据场景
- 明确需重审场景
- 明确阻断项解除条件
- 明确责任人和复审进入条件

## 主持人口径

- 不让“现象描述”替代“证明责任”
- 不让“实现难度”下调验收口径
- 不让“弱场景跑通”替代“首层必需场景成立”
- 不让“单次跑通”替代“稳定重演”
- 不让讨论回退到抽象定义层
