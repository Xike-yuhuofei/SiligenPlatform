# 工程替身系统总纲说明页

## 文档定位

本页是当前阶段所有需求、范围、场景、口径与评审材料的上位约束说明。
它不回答“怎么实现”，只回答这套系统为什么存在、按什么标准判断它是否成立，以及当前阶段应把什么先冻结下来。
本页所冻结的，不是“仿真系统定义基线”，而是**工程替身系统定义基线**。

## 一、对象定位

当前要重新构建的，不是一个“更像设备”的仿真软件，也不是对既有仿真系统的重写或升级。
这里要定义的对象，是一套面向研发流程的**工程替身系统**。

它服务的对象包括：

- 开发过程中的逻辑验证
- 联调过程中的跨边界确认
- 验证过程中的可判定结论形成
- 异常确认过程中的问题定位与恢复判断
- 回归判断过程中的稳定重演与结果对比

这套系统的本质，不是复制设备世界，而是承接研发流程中原本必须依赖真机才能完成的关键判断责任。

## 二、价值标准

这套系统的价值判断标准，不是“像不像真机”，而是它能否承担研发活动中的关键判断。
后续所有需求、边界与裁剪，都应回到以下标准：

- 能否让研发活动中的关键判断尽可能在脱离真机条件下完成
- 能否把问题暴露前移到上机前
- 能否支撑可重复、可比较、可判定的验证闭环

如果某项内容的主要价值只是展示完整性、外观还原度、建模完整度或表现效果，那么它不应自动进入当前阶段的核心目标。

## 三、当前阶段目标

当前阶段处于**需求澄清模式**。
本阶段的首要任务不是求解实现方案，而是先冻结定义。

当前必须先钉住的内容包括：

- 这套系统为什么存在
- 它必须承接什么
- 它明确不承接什么
- 在什么条件下才算成立
- 用什么场景与口径来判断成立

只有这些内容先稳定下来，后续技术讨论才不会被已有实现路径、既有习惯或临时便利性反向塑形。

## 四、基本判断原则

- 以“关键判断是否可尽可能脱离真机完成”为核心，而不是以“工作是否看起来更完整”为核心
- 以“是否形成研发闭环”为核心，而不是以“是否支持某个单点功能”为核心
- 以“是否可用于工程判断”为核心，而不是以“是否更像设备运行”为核心
- 以“是否能够支撑后续自动化验证与回归判断”为核心，而不是以“是否能做演示”为核心

## 五、前提与边界

- 必须从头定义新的工程替身系统，不能以现有在研仿真系统作为需求来源
- 不能让既有能力、既有边界或既有实现结构反向决定真实需求
- 当前阶段不进入实现方案、技术选型、模块划分、排期与资源讨论
- 当前阶段允许形成需求定义、范围边界、成立条件、场景证明集与评审口径
- 若某项内容不能直接服务研发判断责任承接，就不应被默认纳入首层成立条件

## 六、对后续材料的约束

后续材料应持续受本页约束，包括但不限于：

- 需求定义稿
- 非目标边界
- 成立条件稿
- 范围定义稿
- 首层成立条件
- 首层场景证明集
- 验收口径
- 评审执行包

这些材料的目标，不是把系统描述得越来越大，而是逐步回答：

1. 这套工程替身系统在定义上如何成立
2. 首层最小研发闭环由哪些内容构成
3. 用哪些高判别力场景来证明它成立
4. 用什么评审口径来做可判定结论

## 七、当前阶段结论

当前“工程替身系统定义基线”由以下四份文稿共同构成：

- 工程替身系统总纲说明页
- 需求定义稿
- 非目标边界
- 成立条件稿

当前可以明确采用的总纲结论是：

这套系统应被定义为一套面向研发流程的工程替身系统。
它的成立标准，不取决于对设备世界的还原度，而取决于它是否能够承接研发活动中的关键判断责任，并让这些关键判断尽可能在脱离真机条件下完成，进而形成最小研发闭环。

在本阶段，优先级最高的工作是冻结这套定义基线，而不是提前进入实现讨论。
后续若进入范围边界、首层分层、场景证明集或评审口径阶段，均不得回到对象定义层重新打开问题空间。
