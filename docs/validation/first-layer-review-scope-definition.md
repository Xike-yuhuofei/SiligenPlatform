# 范围定义稿

## 文档定位

本稿用于冻结“工程替身系统”在当前阶段的范围边界。
它不回答如何实现这些范围，也不回答哪些内容应在首期版本先做，而是只回答三件事：

- 哪些内容属于系统成立所必须覆盖的**核心范围**
- 哪些内容属于有明确价值但可后续展开的**扩展范围**
- 哪些内容在当前阶段应**明确保留给真机**

本稿基于以下已冻结文稿展开：

- 《工程替身系统总纲说明页》
- 《需求定义稿》
- 《非目标边界》
- 《成立条件稿》

## 一、判断原则

范围边界的判断标准，不是看某项内容“有没有价值”，而是看它缺失时会不会直接动摇这套系统在定义上的成立。

本稿采用以下三条判断原则：

- 若缺失后会直接削弱关键判断尽可能脱离真机完成的能力，则应进入核心范围
- 若缺失后不会否定系统成立，但会明显影响覆盖深度、验证效率或长期收益，则应进入扩展范围
- 若某项判断本质上仍依赖真实设备、真实工艺或真实物理结果，则应明确保留给真机

## 二、核心范围

以下内容属于当前阶段系统成立所必须覆盖的核心范围；若缺失，系统在定义上不足以成立为工程替身系统。

### 1. 研发验证上下文的主动构造

系统必须能够主动进入研发所需的验证上下文，而不是继续依赖真机自然进入某个阶段后再开始判断。

### 2. 支撑研发判断的有效反馈

系统必须能够提供足以支撑研发判断的反馈，使研发能够明确知道为什么能继续、为什么不能继续、当前卡点在哪里、结论是否成立。

### 3. 关键交互、控制链路与操作序列的验证

系统必须能够承接关键交互时序、控制链路和操作序列的验证责任，而不是只支持局部逻辑或单点状态观察。

### 4. 异常构造与恢复路径判断

系统必须能够主动构造异常，并判断异常后的处理、恢复或终止路径是否成立，而不是只承接正常流程。

### 5. 数据化场景组合下的行为确认

系统必须能够在关键参数、配方版本、配置组合或输入条件变化时形成可比较结论，而不是只能运行单一默认路径。

### 6. 稳定重演、比较与判定

系统必须能够支持同一验证场景稳定重复执行，使结论能够重演、比较和复查，而不是依赖一次性现象或临场经验。

## 三、扩展范围

以下内容具有明确价值，但在当前阶段缺失并不会直接否定系统的成立；它们应被视为后续可展开的扩展范围，而不是当前成立门槛。

### 1. 覆盖深度扩展

在已成立的判断责任之上，继续扩展更多业务支路、更多参数组合、更多异常类型和更多恢复路径深度，属于扩展范围。

### 2. 反馈丰富度扩展

在已具备有效反馈的前提下，继续增强反馈维度、观察便利性、对比视角和分析友好性，属于扩展范围。

### 3. 场景规模扩展

在已具备最小研发闭环的前提下，继续增加场景数量、覆盖广度和回归样本规模，属于扩展范围。

### 4. 复用与效率增强

在已满足判断成立的前提下，继续提升场景复用效率、批量验证便利性和结论整理效率，属于扩展范围。

## 四、明确保留给真机

以下内容在当前阶段应明确保留给真机，不纳入工程替身系统的承接范围：

### 1. 最终物理验收

涉及真实物理结果、真实设备状态和最终物理指标确认的判断，应保留给真机。

### 2. 最终工艺效果确认

涉及最终工艺效果、最终制程质量或最终产出效果的结论，应保留给真机。

### 3. 最终性能与极限能力确认

涉及最终性能边界、真实节拍、真实精度极限或真实现场稳定性的结论，应保留给真机。

### 4. 真实现场环境耦合判断

必须依赖真实现场环境、真实硬件条件或真实联机链路才能成立的最终结论，应保留给真机。

## 五、边界使用约束

- 核心范围用于回答“系统成立必须覆盖什么”，不用于回答“首期版本先做什么”
- 扩展范围用于回答“后续可以在哪些方向继续增强”，不用于弱化当前成立门槛
- 真机保留用于明确当前不应由工程替身系统替代的判断，不代表这些内容没有价值
- 后续若进入“核心范围内再分层”，只能在核心范围内部再分首层必需与后置展开，不能重新改写本稿三分边界

## 六、本稿结论

当前可冻结的范围边界结论是：

这套工程替身系统的当前范围，应按“核心范围 / 扩展范围 / 明确保留给真机”三分法固定。
其中，验证上下文构造、有效反馈、关键链路验证、异常与恢复判断、数据化场景对照、稳定重演与判定，属于系统成立所必须覆盖的核心范围；
覆盖深度、反馈丰富度、场景规模和验证效率增强，属于扩展范围；
最终物理验收、最终工艺效果确认、最终性能与极限能力确认、真实现场环境耦合判断，明确保留给真机。
