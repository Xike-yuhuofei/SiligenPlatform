# ADR 存放位置方案对比分析

> **背景**：项目中存在两个位置存放 ADR（架构决策记录），违反 Single Source of Truth 原则：
> - `docs/adr/` - 包含 4 个历史 ADR（adr-001 ~ adr-004）
> - `docs/library/03_decisions/` - 包含新的 ADR 模板和示例
>
> **目标**：选择统一的 ADR 存放位置，明确文档系统架构

---

## 📋 三个方案概览

| 方案 | 描述 | ADR 位置 | docs/ 结构变化 |
|------|------|---------|---------------|
| **方案1** | 统一到 `docs/adr/` | `docs/adr/` | 删除 `library/03_decisions/` |
| **方案2** | 分离活跃和历史 | `docs/adr/` + `library/03_decisions/` | 保持双目录结构 |
| **方案3** | 统一到 `docs/library/03_decisions/` | `docs/library/03_decisions/` | 删除 `docs/adr/` |

---

## 方案1：统一到 `docs/adr/`

### 🎯 方案定义

**将 `library/03_decisions/` 的内容迁移到 `docs/adr/`，删除 `library/03_decisions/`**

**新的文档结构**：
```
docs/
├── library/
│   ├── 00_index.md
│   ├── 01_overview.md
│   ├── 02_architecture.md
│   ├── 04_guides/        # 从03跳到04
│   ├── 05_runbook.md
│   └── 06_reference.md
├── adr/                  # 所有ADR的统一位置
│   ├── adr-001-hexagonal-architecture.md
│   ├── adr-002-namespace-unification.md
│   ├── adr-003-result-pattern.md
│   ├── adr-004-remove-grpc.md
│   ├── ADR-20260111-hexagonal-architecture.md
│   ├── ADR_TEMPLATE.md
│   └── README.md
├── _archive/
└── plans/
```

### ✅ 优势

| 维度 | 详细分析 |
|------|---------|
| **1. Single Source of Truth** | ADR 只有一个权威位置，避免混淆和重复 |
| **2. 符合社区标准** | ADR 社区最佳实践是独立目录（参考 [adr.github.io](https://adr.github.io/)） |
| **3. 历史连续性** | `docs/adr/` 已有 4 个历史 ADR，保持编号和引用连续 |
| **4. 成熟系统** | 已有完整的 README、索引、模板、使用指南（2496行文档） |
| **5. 用户认知符合直觉** | "架构决策记录" → `adr/` 目录，符合命名直觉 |
| **6. 维护简单** | 只需维护一个目录、一个索引、一套模板 |
| **7. 链接清晰** | 所有外部引用指向 `docs/adr/`，无需考虑新旧 |
| **8. 工具集成便利** | 可直接使用标准 ADR 工具（adr-tools, adr-log, ADR GitHub Actions） |
| **9. 可移植性强** | `docs/adr/` 可以独立发布、共享、作为子模块引用 |
| **10. 符合五层架构** | `adr/` 是独立的 TIER 4，符合五层文档系统设计 |

### ❌ 劣势

| 维度 | 详细分析 |
|------|---------|
| **1. 破坏 library/ 编号** | library/ 从 03 跳到 04，编号断层（但可接受） |
| **2. library 不完整** | library/ 缺少决策记录部分，可能被认为"不够全面" |
| **3. 跨目录访问** | 查阅 library 索引时需要跳转到 adr/（但可通过链接解决） |
| **4. SSoT 分散** | 核心知识分散在 library/ 和 adr/ 两个位置 |

### 🎯 适用场景

- ✅ 中小型项目（ADR 总数 < 50）
- ✅ 需要快速决策的项目
- ✅ 团队习惯于 ADR 标准实践
- ✅ 追求简洁的文档结构
- ✅ 有工具化需求（ADR 自动化工具）

## 方案2：分离活跃和历史

### 🎯 方案定义

**`docs/adr/` 存放活跃的 ADR，`docs/library/03_decisions/` 存放已验证的核心 ADR**

**文档结构**：
```
docs/
├── library/
│   ├── 03_decisions/        # 已验证的核心ADR
│   │   ├── ADR-001-hexagonal-architecture.md  (副本)
│   │   ├── README.md
│   │   └── ...
├── adr/                     # 所有ADR（活跃+历史）
│   ├── adr-001-hexagonal-architecture.md      (原始)
│   ├── adr-002-namespace-unification.md
│   └── ...
```

**流动规则**：`adr/` → 经过验证 → 复制到 `library/03_decisions/`

### ✅ 优势

| 维度 | 详细分析 |
|------|---------|
| **1. 信息分层** | 区分"活跃决策"和"核心决策"，降低认知负担 |
| **2. library 完整性** | library/ 包含所有核心知识（包括决策） |
| **3. 渐进式归档** | ADR 可以从 `adr/` 按生命周期流动到 `library/03_decisions/` |
| **4. 受众区分** | 架构师关注 `library/03_decisions/`（核心决策），开发者关注 `adr/`（所有决策） |

### ❌ 劣势

| 维度 | 详细分析 |
|------|---------|
| **1. 违反 SSoT** | ADR 有两个位置，用户困惑："新 ADR 放哪？" |
| **2. 维护成本高** | 需要维护两套索引、两套模板、两个 README |
| **3. 模糊的归档标准** | 什么情况下从 `adr/` 移到 `library/03_decisions/`？<br>• 6个月后？（时间标准）<br>• 被 5 个其他 ADR 引用？（影响标准）<br>• 人工判断？（主观、易遗漏、不可靠） |
| **4. 同步问题** | 如果 ADR 在 `adr/` 被修改，`library/03_decisions/` 的副本是否同步？<br>• 自动同步？（复杂）<br>• 手动同步？（易遗漏）<br>• 不同步？（导致不一致） |
| **5. 用户困惑** | "这个决策应该在 `adr/` 还是 `library/03_decisions/`？" |
| **6. 搜索复杂** | 需要搜索两个目录才能找到所有 ADR |
| **7. 工具不兼容** | 标准 ADR 工具不支持双目录结构 |
| **8. 引用混乱** | 文档中引用 ADR 时需要判断位置：<br>`[ADR-001](../adr/adr-001.md)` vs `[ADR-001](./ADR-001.md)` |
| **9. 重复存储** | 同一个 ADR 在两个位置存在，浪费存储空间 |
| **10. 版本控制复杂** | Git 历史会包含多次复制操作，污染历史记录 |

### 🎯 适用场景

- ✅ 超大型项目（ADR 总数 > 100）
- ✅ 长期维护（10+ 年）的项目
- ✅ 有专门的文档管理员
- ✅ 有明确的自动化归档规则和工具

## 方案3：统一到 `docs/library/03_decisions/`

### 🎯 方案定义

**将所有 ADR 从 `docs/adr/` 迁移到 `docs/library/03_decisions/`，删除 `docs/adr/`**

**新的文档结构**：
```
docs/
├── library/
│   ├── 00_index.md
│   ├── 01_overview.md
│   ├── 02_architecture.md
│   ├── 03_decisions/        # 所有ADR的统一位置
│   │   ├── adr-001-hexagonal-architecture.md
│   │   ├── adr-002-namespace-unification.md
│   │   ├── adr-003-result-pattern.md
│   │   ├── adr-004-remove-grpc.md
│   │   ├── ADR-20260111-hexagonal-architecture.md
│   │   ├── ADR_TEMPLATE.md
│   │   └── README.md
│   ├── 04_guides/
│   ├── 05_runbook.md
│   └── 06_reference.md
├── _archive/
└── plans/
```

**删除**：`docs/adr/` 目录

### ✅ 优势

| 维度 | 详细分析 |
|------|---------|
| **1. library 完整性** | ⭐⭐⭐⭐⭐ library/ 包含**所有**核心知识（概览、架构、决策、指南、排障、参考），成为真正的 Single Source of Truth |
| **2. 符合知识库定义** | ADR 是"稳定、经过验证的知识"，符合 library/ 的定位（vs _archive 的"过程文档"） |
| **3. 统一索引体验** | 用户只需要查阅 `library/00_index.md` 和 `library/000_index.md`，无需跨目录 |
| **4. 符合五层系统逻辑** | library/ 是 TIER 1（权威知识库），ADR 属于权威知识，应该在 TIER 1 |
| **5. 元数据一致性** | 所有 library/ 文档使用相同的元数据规范，便于管理和审查 |
| **6. 概念清晰** | "架构决策"是核心知识的一部分，不是独立的文档类型 |
| **7. 减少 docs/ 根目录** | docs/ 下保留核心目录（library, _archive, plans, adr），结构更清晰 |
| **8. 查阅效率** | 新同学只需记住 library/ 一个位置即可找到所有核心文档 |
| **9. SSoT 统一** | 所有核心知识统一在 library/，符合 Single Source of Truth 原则 |

### ❌ 劣势

| 维度 | 详细分析 |
|------|---------|
| **1. 违反 ADR 社区标准** | ADR 社区（[adr.github.io](https://adr.github.io/)）推荐独立目录，便于工具集成和跨项目复用 |
| **2. 丧失可移植性** | `docs/adr/` 可以独立发布、共享、工具化，嵌入 library/ 后失去这个优势 |
| **3. 破坏历史连续性** | 已有外部链接引用 `docs/adr/adr-001.md`，迁移后所有链接失效 |
| **4. 迁移成本高** | 需要更新 `docs/adr/README.md` 中的所有内部引用、索引、脚本（2496行文档） |
| **5. 与 TIER 定义冲突** | 五层文档系统定义 `adr/` 为独立 TIER 4，移入 library/ 后需要重新定义架构 |
| **6. 可扩展性受限** | 如果未来 ADR 数量爆炸（>100），library/ 会变得臃肿，而独立 `adr/` 可以独立扩展 |
| **7. 工具不兼容** | 标准 ADR 工具（adr-tools、adr-log）默认查找 `docs/adr/`，需要配置或修改 |
| **8. 混淆文档类型** | library/03_decisions/ 中混合了"文档索引"（README.md）和"内容文档"（ADR-001.md），层级不够清晰 |
| **9. 命名冲突风险** | 如果未来有其他类型的决策记录（如 Design Decision Records），难以扩展 |
| **10. 失去专业化管理** | ADR 有其独特的生命周期、模板、评审流程，混入 library/ 后难以独立管理 |

### 🎯 适用场景

- ✅ 追求 library 绝对完整性的项目
- ✅ 不需要 ADR 工具化的项目
- ✅ 内部文档，无需跨项目共享
- ✅ ADR 数量预期不会很多（< 50）

## 🤔 核心矛盾分析

### 方案1 vs 方案3 的根本分歧

**问题：ADR 的本质属性是什么？**

| 观点 | ADR 是... | 应该放在... | 类比 |
|------|----------|------------|------|
| **方案1 观点** | **独立的文档类型**<br>（类似代码、测试、配置） | `docs/adr/`<br>（独立目录，便于专业化管理） | `tests/` 独立于 `src/` |
| **方案3 观点** | **核心知识的一部分**<br>（类似概览、架构、指南） | `docs/library/03_decisions/`<br>（知识库的有机组成部分） | API 文档在 docs/ 中 |

### 两种观点的哲学基础

**方案1 的哲学 - 方法论优先**：
- ADR 是**工作方法**，需要工具化、标准化
- 参考 ADR 社区实践（独立目录、adr-tools）
- 便于跨项目共享和复用
- **类比**：
  - `tests/` 目录独立于 `src/`，尽管测试也是"项目质量"的一部分
  - `.github/` 目录独立，尽管 CI/CD 也是开发流程的一部分

**方案3 的哲学 - 知识统一优先**：
- ADR 是**核心知识**，需要统一管理
- Single Source of Truth 意味着所有核心知识在 library/
- 降低用户认知负担（只需记住 library/）
- **类比**：
  - 设计文档、API 文档、用户手册都在 docs/ 中
  - 不会为每种文档类型创建独立目录

**推荐理由**：

1. **library 真正成为 Single Source of Truth** - 所有核心知识统一管理
2. **用户体验更简洁** - 只需记住 library/ 一个位置
3. **查阅效率更高** - 新同学无需跨目录查找

**适用条件**：
- ✅ **library 的绝对完整性是最高优先级**
- ✅ 不需要 ADR 工具化（仅手动管理）
- ✅ 愿意承担高迁移成本（更新 2496 行文档）

### 不推荐：方案2（分离活跃和历史）

**不推荐理由**：

1. **违反 Single Source of Truth** - ADR 有两个权威位置
2. **维护成本过高** - 需要维护两套系统
3. **归档标准模糊** - 缺乏明确的自动化规则
4. **同步问题复杂** - 两个位置的 ADR 如何保持一致
5. **用户困惑** - 不清楚应该在哪个位置查找或创建

**唯一适用场景**：
- ⚠️ 超大型项目（ADR > 100）
- ⚠️ 有专门的文档管理员
- ⚠️ 有完善的自动化归档工具

## 🔑 关键决策点

选择方案前，请回答以下问题：

### 问题1：Single Source of Truth 的定义范围？

**A. 所有核心知识必须在 library/ 中**
→ 选择方案3

**B. 核心知识可以分散在不同层级（library/、adr/、plans/）**
→ 选择方案1

---

### 问题2：ADR 工具化的需求程度？

**A. 强烈需要（adr-tools、ADR GitHub Actions、自动化生成）**
→ 选择方案1

**B. 不需要（仅手动管理，无工具化计划）**
→ 方案1 和方案3 都可

---

### 问题3：已有投资的保留优先级？

**A. `docs/adr/` 的 2496 行文档和成熟系统必须保留**
→ 选择方案1

**B. 愿意重写和迁移，追求理想架构**
→ 选择方案3

---

### 问题4：迁移成本的可接受度？

**A. 低（只移动 2 个文件，最小化变更）**
→ 选择方案1

**B. 高（移动 4+ 个文件，更新所有引用和链接）**
→ 选择方案3

---

### 问题5：五层文档系统的稳定性？

**A. 架构已定义，不应轻易修改**
→ 选择方案1

**B. 架构可以优化，为了更好的 SSoT**
→ 选择方案3
