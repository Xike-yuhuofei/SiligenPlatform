## 您的优化提示词（Claude Code）：

请作为Siligen运动控制器项目的审计专家，对提供的实施计划进行全面审计。基于项目的六边形架构规范和工业安全要求，执行以下系统化审计：

## 一、审计目标
创建一个可重用的审计模板，用于评估未来所有实施计划的完整性、合规性和可行性。

## 二、项目上下文参考
- **项目结构**：参考 `D:\Projects\SiligenSuite\control-core\.xike\docs\project-structure-and-tech-stack.md`
- **架构规则**：检查 `.claude/rules/` 目录下的所有规则文件，特别是：
  - `HEXAGONAL.md` - 六边形架构依赖方向
  - `DOMAIN.md` - 领域层约束（无动态内存、无STL容器、无IO/线程）
  - `INDUSTRIAL.md` - 工业安全约束（实时性、紧急停止、阀门安全）
  - `NAMESPACE.md` - 命名空间规范
  - `PORTS_ADAPTERS.md` - 端口接口设计
  - `REALTIME.md` - 实时系统约束
  - `EXCEPTIONS.md` - 例外抑制机制

## 三、审计步骤

### Step 1：计划完整性检查
1. **结构完整性**：检查计划是否包含以下标准部分：
   - 目标/执行摘要
   - 架构设计描述
   - 技术栈说明
   - 任务分解（Task/Phase）
   - 详细实施步骤
   - 代码示例
   - 测试验证策略
   - 风险评估与缓解
   - 成功标准和验证清单

2. **文件清单验证**：
   - 是否列出所有需要创建/修改的文件路径？
   - 文件路径是否符合项目命名约定？
   - 目录结构是否遵循 `src/domain/`, `src/application/`, `src/infrastructure/` 分层？

### Step 2：架构合规性审计
1. **六边形架构验证**：
   - 依赖方向：Infrastructure → Application → Domain（单向）
   - Domain层是否保持零技术依赖？
   - 是否通过Port接口访问Domain层？
   - 是否存在反向依赖或循环依赖？

2. **命名空间合规性**：
   - 根命名空间是否为 `Siligen`？
   - 分层命名空间是否符合：
     - `src/domain/**` → `Siligen::Domain::*`
     - `src/application/**` → `Siligen::Application::*`
     - `src/infrastructure/**` → `Siligen::Infrastructure::*`
     - `src/adapters/**` → `Siligen::Adapters::*`
     - `src/shared/**` → `Siligen::Shared::*`
   - 头文件中是否禁止 `using namespace` 指令？

3. **领域层约束检查**：
   - 是否避免使用 `new`、`delete`、`malloc`？
   - 是否避免使用 `std::vector`、`std::map`（除了motion模块的特例）？
   - 如果使用STL容器，是否添加了 `CLAUDE_SUPPRESS` 抑制注释？
   - 公共API是否标记为 `noexcept`？
   - 是否避免使用 `std::cout`、`std::thread`？

### Step 3：工业安全约束审计
1. **实时性约束**：
   - 控制循环中是否避免动态内存分配？
   - 是否避免日志记录？
   - 是否避免STL容器？
   - 是否使用 `std::scoped_lock` 进行线程安全？

2. **紧急停止合规性**：
   - 是否禁止阻塞操作？
   - 是否禁止条件执行？
   - 是否禁止日志记录？
   - 是否保证10ms内返回？

3. **硬件隔离**：
   - Domain层是否避免直接包含硬件头文件？
   - 是否通过 `IHardwareController_port` 访问硬件？
   - 适配器是否实现单一端口接口？

### Step 4：实施可行性审计
1. **代码示例验证**：
   - 提供的代码示例是否完整且可编译？
   - 是否包含必要的头文件包含？
   - 函数签名是否完整？
   - 错误处理是否使用 `Result<T>` 模式？

2. **CMake配置检查**：
   - 是否更新 `CMakeLists.txt`？
   - 是否添加正确的库依赖？
   - 测试目标是否配置正确？

3. **测试策略评估**：
   - 是否包含单元测试计划？
   - 测试用例是否覆盖关键路径？
   - 是否包含集成测试和性能测试？
   - 测试命名是否符合 `TEST(ClassName, MethodName_Scenario_ExpectedResult)` 格式？

### Step 5：风险管理和测试覆盖审计
1. **风险识别完整性**：
   - 是否识别技术风险、集成风险、性能风险？
   - 风险评估是否包含概率、影响、缓解策略？
   - 是否有回滚/恢复计划？

2. **测试覆盖检查**：
   - 是否针对每个功能点设计测试用例？
   - 是否包含边界条件测试？
   - 是否包含错误处理测试？
   - 是否包含性能基准测试？

### Step 6：例外机制审计
1. **抑制注释检查**（如果适用）：
   - 是否有充分的理由说明（至少20字符）？
   - 是否包含批准人和日期？
   - 规则是否在 `ALLOWED_SUPPRESSIONS` 列表中？
   - 禁止抑制的规则（如硬件隔离、紧急停止）是否被错误抑制？

## 四、输出要求

### 1. 审计报告文档（Markdown格式）
- **执行摘要**：总体审计结论、关键发现、优先级（P0/P1/P2）
- **详细发现**：按审计步骤分类的问题列表
- **合规性评分**：架构合规性、安全合规性、实施可行性评分
- **建议优先级**：高优先级、中优先级、低优先级

### 2. 直接修复建议
- 具体的代码修改建议
- 文件更新建议
- 架构调整建议
- 每个建议包含具体的代码示例

### 3. 验证命令清单
- 构建验证命令
- 测试执行命令
- 规则检查命令
- 格式化检查命令

### 4. 规则违反清单
- 表格格式：规则名称、文件路径、行号、违规内容、严重程度
- 引用具体的规则文件（`.claude/rules/*.md`）
- 说明如何修复

## 五、验证计划
1. **架构验证**：运行 `scripts/analysis/arch-validate.ps1`
2. **规则检查**：对每个规则文件使用 `rg` 命令验证
3. **构建验证**：确保所有代码示例可编译
4. **测试验证**：确保测试计划可执行

## 六、成功标准
- 所有架构违规被识别并提出解决方案
- 所有安全约束被验证
- 实施计划中的技术风险被全面评估
- 测试策略覆盖所有关键功能点
- 审计报告包含可操作的修复建议

## 使用说明

1. **复制整个提示词**：从"您的优化提示词（Claude Code）："开始复制
2. **在Claude Code中粘贴**：在对话中输入 `/clear` 清空上下文，然后粘贴此提示词
3. **提供审计对象**：在提示词后附加需要审计的实施计划文档内容
4. **执行审计**：Claude Code将按步骤执行审计并生成报告

## 示例命令

```bash
# 清空上下文
/clear

# 粘贴此提示词，然后附加要审计的计划文档
# Claude Code将自动执行审计流程
```

## 输出示例

审计完成后，您将获得：
1. `audit-report-YYYY-MM-DD.md` - 完整的审计报告
2. `fix-suggestions.md` - 具体的修复建议
3. `validation-commands.txt` - 可执行的验证命令清单
4. `rule-violations.csv` - 规则违反清单

## 注意事项

1. **环境要求**：确保已安装 ripgrep (rg) 用于规则检查
2. **权限要求**：需要读取 `.claude/rules/` 目录的权限
3. **时间预估**：完整审计通常需要15-30分钟
4. **审计深度**：根据计划复杂度调整审计深度
